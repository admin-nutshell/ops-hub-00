## What changed

New file `scripts/t98-gate.sh` — reads `monitor-e2e-pipeline.yml`'s
(T-98's) own already-recorded run history via `gh run list`/`gh run view`
and fails loudly unless the most recent **real** (non-skip) downstream-
pipeline check both passed and is recent enough. Wired into
`prod-deploy.yml` as a new **step 0** — a pre-flight check that runs
before any prod mutation (image patch/start), gating every prod
promotion.

Full design rationale — including the live staleness finding that shaped
the algorithm, the hard-block-vs-flag fork, and the applies-to-every-
promotion fork — is written into the script's own header comment and
`DECISIONS.md`'s 2026-07-16 "T-124" entry; not duplicated here.

---

## Why this can't false-positive on a genuinely healthy, actively-monitored pipeline

If T-98 has taken a real reading within the last 24h and it passed, this
gate passes immediately — it adds zero new failure surface beyond "can we
read GitHub's own Actions API," which `prod-deploy.yml` already implicitly
trusts (it dispatches via the same API).

## Why this can (and, right now, does) catch a real gap

Not hypothetical — confirmed against live data at build time. See
"Verification" below and the script's own header for the full evidence:
T-98 has not taken a real reading of the downstream ticket pipeline in
~74 hours (as of this writing), because `ops-hub-staging` has apparently
been left running through today's heavy deploy/verification activity,
which trips T-98's own `staging_guard` (T-111) every 6-hourly cycle. This
gate correctly fails right now, against real repo state — not a contrived
test case.

---

## Rollback path (defined BEFORE dispatch, per team quality bar)

This change is CI-workflow-only — it adds a pre-flight gate, it does not
change what gets deployed once the gate passes, and (being a pre-flight
check) it runs before any prod mutation happens at all. If the gate itself
is somehow wrong (e.g. a transient GitHub API blip makes it fail when T-98
is genuinely healthy and fresh), the rollback is: revert this PR's commit,
or re-run `prod-deploy.yml` once the transient condition clears — nothing
about the application's runtime state is touched by this change, so there
is no deployed state to unwind. Mean rollback time: well under 15 minutes.

If the gate correctly blocks on a real T-98 problem, the remediation is
named directly in the gate's own `::error::` output: stop
`ops-hub-staging` (if it's the staleness cause) and either wait for the
next 6-hourly cycle or manually dispatch `monitor-e2e-pipeline.yml` with
`mode=live` for a fresh reading, then re-run `prod-deploy.yml`.

## Reviewer decision point (not resolved in this PR)

A stale T-98 signal — the common real-world case right now, per the
staleness finding above — is caused by an ORTHOGONAL condition
(`ops-hub-staging` left running) rather than production actually being
unhealthy. This gate hard-blocks with no bypass input, so an urgent prod
hotfix would also be blocked if T-98 happens to be stale at that moment,
even with nothing wrong downstream. This is a deliberate, documented
choice (see `scripts/t98-gate.sh`'s header, design fork #2), not an
oversight — but T-123's own QA review already logged wanting an
emergency-bypass runbook for the sibling `deploy-health-gate.sh`.
Reviewers should consciously decide whether "no bypass on the only
prod-promotion path" is acceptable as shipped, or whether a documented
override (e.g. a `workflow_dispatch` input) belongs in this PR before
merge rather than as a follow-up.
