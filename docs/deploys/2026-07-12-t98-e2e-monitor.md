## Why this exists

Three incidents this year (T-71, FQ-69, FQ-70) were all "a green health
signal while the real path was 100% broken." T-97 closed the internal-auth
*hop* specifically (`/health/litellm-internal` exercises the same credential
resolution `classifyTicket()` uses) but never dispatches a real ticket
through Inngest, never exercises `ticket-respond`, never checks a DB state
transition, and never checks a LangFuse trace. T-98 closes that remainder:
real event dispatch → triage → (respond, once SC9 lands) → DB state → trace.

## Deviation from the handoff prompt (documented, not silently corrected)

The task prompt asserted a "T-51/earlier precedent" of creating FreeScout
test conversations via its REST API. Checked before building anything —
**false**: `docs/adr/0003-freescout-response-writeback.md` Option C
explicitly rejected the REST API as a delivery path (FreeScout's Api module
is disabled-by-default; no paid module was purchased), and T-51's own "E2E
validation" (WORK.md, 2026-07-03) sent a real inbound *email*, not an API
call — no REST-API precedent exists anywhere in this repo. SC9's own text
("create via the FreeScout UI... a raw conversations INSERT is rejected") is
the authoritative, load-bearing spec here and was followed literally instead.
Consequence: the actual FreeScout conversation + sentinel-ticket provisioning
(SC9's full sequence) requires interactive FreeScout admin UI access this
session does not hold a password for, and is sequenced after the DB role
exists (the sentinel INSERT doubles as the role's own positive test) — so it
is documented below as a post-migration runbook, not executed this session.

than erroring, retrying, or writing anywhere — no customer impact; cost is
bounded to one real `classifyTicket()` LLM call, the same order of magnitude
as any single real prod ticket already processes.

## Inngest event-key capability — verified, not assumed

Confirmed against Inngest's own docs: multiple named event keys per
environment are supported (dashboard "+Create Event Key"), independently
renameable and revocable — a dedicated monitor-only key is a real pattern.
**Not confirmed:** whether Inngest's "filter by event name" key control is a
true access-control restriction (vs. a dashboard search filter) — the docs
describe the feature without specifying enforcement. FQ-75 asks the founder
to check for this option while minting the key; if absent, the real safety
boundary is the DB-role-level tenant hardcoding, not the event key itself
(named honestly in DECISIONS.md, not glossed over).

## Honest scope note (SC10)

This monitor does NOT and CANNOT exercise the FreeScout-polling/intake stage.
`freescout-poller.ts` stamps `POLLING_TENANT_ID` on every conversation it
matches (status=1, state=2) with no mailbox filter — any poller-visible
synthetic conversation would land in the REAL prod tenant by construction,
recreating exactly the hole Amendment A1/Ruling 2 closed. Coverage starts at
Inngest event dispatch, the same point `sweepNewTickets`/the poller would
otherwise hand off to. Intake stays covered by T-97 + the FQ-69-era manual
diagnostics.

## Rollback path (defined before dispatch, per team quality bar)

- **Migration:** additive-only (new role, new policies on an existing table).
  Rollback = `DROP POLICY` x3 + `DROP ROLE e2e_monitor` (or simply never set
  a password / revoke the GitHub secret — the role is inert without one).
  No existing role, grant, or policy is modified.
- **Workflows:** purely additive new files. Rollback = disable the
  `schedule:` trigger on `monitor-e2e-pipeline.yml` (or delete the file) if
  it ever produces false positives — no other workflow or app behavior is
  touched.
- **Credentials:** `E2E_MONITOR_DB_URL` / `E2E_MONITOR_INNGEST_KEY` are both
  new, dedicated secrets — revoking either independently (delete the GitHub
  secret, or delete the Inngest event key / reset the role's password) fully
  disables the monitor without touching any other credential.
- Mean rollback time target < 15 minutes; very low rollback risk (nothing
  existing is modified).
