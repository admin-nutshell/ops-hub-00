## Context

### The failure class, stated once

The `live-eval-gate` grades each product-eval test with an `llm-rubric` assertion: a **judge model** reads the target's output against a written rubric and returns a structured verdict `{pass: boolean, score: 0..1, reason: string}`. Promptfoo then decides the assertion's pass/fail. **Because each assertion carries an explicit `threshold: 0.8`, promptfoo decides pass/fail by `score >= 0.8` and discards the grader's own `pass` boolean.** That single design choice is the whole bug:

```
grader returns {pass: true, score: 0.75, reason: "low is not a listed fail condition — minor deviation, not a hard failure"}
  → threshold 0.8 set → promptfoo computes 0.75 >= 0.8 = FALSE
  → assertion FAILS
  → the gate blocks the PR — for a reason the grader itself explicitly disagreed with
```

This conflates two categorically different questions inside one numeric comparison:

1. **Objective contract conformance** — is the output valid JSON? is `urgency` inside the closed enum? is a real outage over- or under-escalated? These are *hard* pass/fail facts. They MUST fail the gate, always.
2. **Subjective nuance within the rubric's own stated tolerance** — e.g. "normal vs low" on a genuinely-borderline vague ticket, where the rubric *itself* says "'low' is a tolerable read" and does *not* list it as a fail condition.

A single 0.8 threshold cannot separate these. It lets a subjective deduction the rubric already forgave (a 0.05 dock below the line) block a PR exactly as hard as an objective contract violation. That is the class.

### The FQ-77 instance — proximate fix shipped, systemic finding banked

FQ-77 (Sprint 12) is the concrete instance. T-105's **real** triage prompt-injection security fix was built and proven, yet its merge was blocked — not by any defect, but because landing it cleanly perturbed a *baselined, untouched-by-the-PR* case (g), a vague low-signal ticket, from `normal` to `low`. Case (g)'s **own written rubric** calls `low` "a tolerable read" and omits it from the fail list; the grader returned `pass: true, score: 0.75` with the verbatim reason that `low` is not a failing value — and the hard 0.8 threshold overrode that `pass: true` anyway (DECISIONS.md 2026-07-12, primary-source grader transcript run `29215571461`).

The **proximate** fix (PR #439) was a one-line calibration: append the house idiom "(Do not fail solely on a normal-vs-low judgement call.)" to case (g)'s fail line — verbatim-present already in sibling cases (i)/(j)/(m)/(p). That fixed *that case*. But the Evals Lead **banked the general finding separately** (DECISIONS.md 2026-07-12): this is a property of *every* borderline case in the suite, not a (g) defect. Per-run `llm-rubric` grading carries variance, and the hard threshold can override the grader's own `pass:true` on any genuinely-borderline case — and it can therefore **block a genuinely correct change** (it did — a real security fix) for a reason the grader itself disagrees with. FQ-77's whole point was *not* to loosen strictness; it was that the gate's own mechanism can be wrong about a correct change. That systemic class is this ADR's subject.

### Two distinct sub-phenomena — name them, because they demand different mechanisms

The banked finding folds together two things that a durable fix must treat separately:

- **(P1) Threshold-vs-rubric-tolerance misalignment (the presenting case).** The grader returns an *honest, stable* `pass: true, score: 0.7–0.75` — it is *deliberately* docking a few points for a nuance its own rubric forgives, while explicitly declaring `pass`. This is **not noise**: re-running the grader returns the same ~0.75 for the same reason. Case (g) is exactly this.
- **(P2) Per-run grader variance (the secondary phenomenon).** For a case whose *true mean* score sits near the threshold, individual runs scatter — one run draws 0.83 (pass), the next 0.78 (fail), on identical inputs. Corroborated in FQ-77: a ticket-**respond** case regressed in one live-gate run although T-105's PR touched nothing in respond's surface (DECISIONS.md 2026-07-12, Tech Lead).

The discriminator that drives the whole decision: **(P1) is the case that actually blocked the security fix, and multi-sample grading does not fix it** — averaging five honest 0.75s is still 0.75, still below 0.8, still a false block. Multi-sample only addresses (P2). A durable fix must solve (P1) first, and may address (P2) as a bounded secondary concern.

### Scope — this ADR is the grader-robustness class ONLY

This ADR addresses the **grader-robustness class** (P1 primarily, P2 secondarily) and nothing else. Explicitly **out of scope**, and NOT to be folded in:

- **Eval-coverage growth toward ADR-0007 §5.4's ≥ 20-tests-per-eval target** (currently N≈16/13/14). That is a separate, additive, opportunistic hygiene arc (WORK.md Sprint 13 flagged carries) — it is a *different* answer to *"is the gate meaningful?"* (more samples) than this ADR's answer to *"does the gate mis-decide borderline cases?"* (better per-assertion decision logic). Growing test counts also incidentally dilutes per-case flakiness, but it does not fix (P1) at all, and this ADR does not depend on it.
- Any other open carry (staging `LITELLM_URL` re-pin / T-107, the T-98 monitor-threshold decision, the provider-credential-divergence class). None are grader-robustness.

### Constraints that bound the design

- **Provider-neutral** (CLAUDE.md; ADR-0007 §5.3): the grader is a swappable LiteLLM alias (`grader ≠ target`). The mechanism must read only fields *any* grader returns; it must not assume a specific model, nor tune to one model's scoring distribution.
- **Preserves drop-don't-weaken** (standing norm; FQ-77 corollary): the mechanism must NOT become a back-door for softening the gate. Over-escalation, malformed output, and out-of-enum results must still fail regardless of the mechanism. This is the load-bearing constraint — §(b) below is dedicated to why the chosen design cannot be gamed into a rubber stamp.
- **Free-tier-first** (CLAUDE.md; ADR-0007 §4 budget ceiling ~$5 CAD/mo expected, $10 CAD/mo hard cap): the mechanism must not blow the gate's cost/latency budget. This is the constraint that decides *against* multi-sample as the default (§(c)).
- **Baseline-relative gate** (ADR-0007 §5.4b): the gate blocks on *regression vs the last green baseline*. The mechanism changes the per-assertion pass/fail that feeds that comparison, so the baseline must be re-captured under the new mechanism before it enforces (ADR-0007 §5.5 analog).
- **Never push to main; all PRs** (non-negotiable #5). The eventual build modifies a shared merge-blocking safety net → §(d) governance boundary.

## Decision

**Adopt Option 3 (Hybrid) — "honor the grader's own `pass` verdict within a bounded borderline band" — as the durable grading mechanism, with two guardrails and one companion hardening:**

1. **Honor-`pass` in the borderline band.** For an `llm-rubric` assertion, when the grader returns `pass: true` with a `score` in the band **[floor, threshold)** (i.e. below 0.8 but not far below), the assertion **passes** — deferring to the grader's own structured verdict instead of failing on the raw number. At/above the threshold it passes as today; this only changes the borderline band.
2. **A hard sanity floor (guardrail 1).** Below the floor, the assertion **fails regardless of `pass`**. The floor is set *strictly below* the observed (P1) borderline band (0.7–0.75) — e.g. **0.6** — so honor-`pass` can never rescue a genuinely-low score, and a grader that emits `pass: true, score: 0.2` (a broken or rubber-stamping grader) still fails. Value is calibration-set at build time against the suite's real borderline distribution.
3. **Never-override-a-fail asymmetry (guardrail 2).** Honor-`pass` only ever resolves the band **in the direction the grader already points**. A grader `pass: false` **always fails**, at any score. The mechanism can turn a borderline *near-miss the grader forgave* into a pass; it can never turn a *failure the grader declared* into a pass.
4. **Companion hardening — split objective contract checks out of the rubric into deterministic assertions.** Move "valid single JSON object" and "`urgency` ∈ {critical,high,normal,low}" from the `llm-rubric` prose into deterministic promptfoo assertions (`is-json` / a small `javascript` enum check) that are **AND-ed** with the `llm-rubric`. Objective contract violations then hard-fail *outside* any grading mechanism — so honor-`pass` provably operates only on the subjective-nuance dimension. This is what makes §(b) airtight rather than merely canary-backstopped.

**Multi-sample grading is NOT adopted as the default** — it fails to fix the presenting (P1) case and it breaks the cost budget (§(c)). It is **retained only as an optional, calibration-gated, per-case escalation** for the narrow (P2)-prone cases (true-mean-near-threshold, high-variance), where the extra judge cost is paid on those few cases only, not on every gate run.

Build is deferred to Sprint 14; this section is the design of record.

### 1. Problem statement

See Context. In one line: **the gate decides `llm-rubric` pass/fail by `score >= 0.8` and discards the grader's own `pass` boolean, so an honest sub-threshold-but-passing borderline verdict (which the rubric's own tolerance forgives) blocks a PR exactly as hard as a real contract violation — and it blocked a correct security fix. Honor the grader's structured verdict in the borderline band; keep objective failures hard and grader-independent.**

### 2. Options considered

#### Option 0 — Do nothing (keep per-case prose calibration, PR #439 style)

On each new borderline collision, hand-append the house idiom "(Do not fail solely on a normal-vs-low judgement call.)" to that case's fail line, as PR #439 did for case (g). **Rejected as the durable answer.** It is the status quo whose ceiling FQ-77 declared: it is reactive (a correct change is blocked first, calibrated after), per-case (every borderline case needs its own hand-tuned prose), and — critically — each such calibration touches the **shared merge-blocking safety net**, so each one needs the user's own direct authorization (Sprint 12 §5.1). That is a governance round-trip *per borderline case, forever*. It also does not touch (P2) variance at all. Kept only as the interim reality until the build lands, and the idiom remains useful as grader-shaping prose thereafter.

#### Option 1 — Multi-sample grading (grade N times, aggregate) — **REJECTED as default; retained as optional per-case escalation**

Grade each test's single target output **N times** with the judge and aggregate (mean, or majority vote on the `pass` boolean).

- **Pro:** directly attacks (P2) per-run variance — a case whose true mean is above the threshold stops flapping because N draws average out the scatter. Conceptually clean.
- **Con — it does not fix the presenting (P1) case.** Case (g)'s grader returns an *honest, stable* 0.75; averaging five 0.75s is 0.75, still below 0.8, still a false block. Multi-sample fixes the *secondary* phenomenon while leaving the *primary* one — the one that actually blocked the security fix — untouched. Any mechanism that doesn't fix (P1) is disqualified as the primary.
- **Con — it breaks the cost/latency budget (§(c)).** The gate's cost is judge-dominated (ADR-0007 §4: judge ≈ $0.027 of the ≈ $0.03 per full-suite run). N=3 judge-samples ≈ **triples** the dominant cost to ≈ $0.08/run and pushes the monthly from ≈ $5.50 CAD toward ≈ **$15 CAD at 130 runs/month — over the $10 CAD hard cap** — and adds 2× judge latency to *every* prompt-touching PR. Paying that on every case to fix a phenomenon that affects a handful of borderline cases is the wrong trade.
- **Disposition:** **Rejected as the default.** **Retained as an optional, calibration-gated, per-case escalation** (`multiSample: N` opt-in on the specific cases empirically shown to be (P2)-prone), so the cost is bounded to those few cases. This is the honest home for multi-sample: a targeted tool for genuine near-threshold variance, not a blanket tax.

#### Option 2 — Margin-based grading (blanket tolerance band) — **REJECTED**

Pass if `score >= (threshold − margin)` — i.e. lower the effective bar to (say) 0.7 for all cases.

- **Pro:** trivially simple; one config number; fixes (P1) mechanically (0.75 ≥ 0.7).
- **Con — it is a back-door gate-loosening and violates §(b).** A blanket margin passes a case **the grader itself failed**: if the grader returns `pass: false, score: 0.72` on a genuine over-escalation, a 0.7 margin passes it anyway. That substitutes an arbitrary looser *number* for the gate's judgement, exactly the "loosened to land the fix" appearance a drop-don't-weaken shop cannot afford (FQ-77's whole point was *not* to loosen strictness). The margin has no principled value — it is a guess, and every borderline case that lands just outside it reopens the same problem one notch down.
- **Disposition:** **Rejected.** The distinction from Option 3 is the crux: honoring the grader's `pass` boolean defers to the *rubric's own structured verdict*; a blanket margin overrides a *number* with a looser number and can flip a grader-declared `fail` to pass. The former is faithful; the latter is a rubber stamp.

#### Option 3 — Hybrid: honor grader `pass` in a bounded band + deterministic objective checks — **CHOSEN**

Defer to the grader's own `pass` boolean in the borderline band [floor, threshold), floored below so a low score always fails, asymmetric so a grader-`fail` always fails, and with objective contract checks moved to deterministic assertions outside the rubric (Decision §§1–4 above).

- **Pro:** fixes (P1) directly — case (g)'s `pass: true, score: 0.75` now passes because the grader said so and 0.75 ≥ floor. **Zero added cost/latency** — it reads the `pass` field the grader *already returns* (FQ-77's transcript is the proof: `pass:true, score:0.75` was already in the response; the threshold merely threw the boolean away). Provider-neutral by construction — it reads structured fields any grader emits and makes no model-specific assumption. **Cannot be gamed into a rubber stamp** — the floor, the never-override-a-fail asymmetry, and the deterministic objective split together mean the mechanism's *entire* behavioral delta is "stop failing a borderline case the grader explicitly passed" (§(b)). Reduces the per-case prose-idiom treadmill (Option 0) to a structural rule.
- **Con / accepted:** the gate now trusts the grader's `pass` boolean in the band, so it leans harder on the anti-rubber-stamp guards (ADR-0007 §5.2 canaries + §5.3 grader≠target) — which already exist and are already required. Sub-threshold-but-passing quality drift is now, *by definition*, within the rubric's stated tolerance; tightening it becomes a rubric-authoring action (monthly coverage review), not a gate-mechanism concern — see Consequences. Needs a one-time floor/band calibration (build task). The deterministic-split companion is a mechanical rewrite of how existing assertions are expressed (not new coverage), touching all three evals.

### (a) Provider-neutral — EXIT CRITERION

The mechanism reads only the **structured fields promptfoo already parses from any `llm-rubric` grader's response** — `pass` and `score`. Those fields are part of the `llm-rubric` contract, not a property of any one model; the grader is a swappable LiteLLM alias (`grader ≠ target`, ADR-0007 §5.3), and today's grader is already a different alias from the target under test. The floor and band are **numeric config, model-independent** — they are calibrated against the *suite's* borderline distribution (a property of the rubrics), not against a model's idiosyncratic scoring curve; a grader swap re-runs the same green-baseline capture (ADR-0007 §5.5) that every grader change already requires, and does not require re-deriving the mechanism. The deterministic companion assertions (`is-json`, enum check) are **pure code with no model at all**. Nothing in the design assumes Claude, or any specific provider, anywhere. **Met.**

### (b) Preserves drop-don't-weaken — EXIT CRITERION (the load-bearing one)

The mechanism collapses to one rule: **an `llm-rubric` assertion passes ⟺ `grader.pass === true` AND `score ≥ floor`.** The old `threshold: 0.8` is *subsumed*, not a surviving boundary (this is why the build must relocate it out of the per-assertion config — see §(c) and the review's C1). The behavioral delta versus today is therefore **bidirectional, and both directions are drop-don't-weaken-safe:**

- It **stops failing** a borderline assertion whose grader returned `pass: true` with a `score` in the band [floor, 0.8) — the FQ-77 (P1) case. (A loosening in exactly one narrow, rubric-sanctioned spot.)
- It **starts failing** a high-score assertion (score ≥ 0.8) whose grader returned `pass: false` — today's `score >= 0.8` rule passes that; honor-`pass` fails it. (A *tightening*: the gate now respects a grader's explicit `fail` even at a high score, instead of overriding it with the number.)

The net is not "strictly narrower" — it is "decide by the grader's structured verdict, floored." Concretely, each protection that FQ-77 insisted must survive, and *why it survives*:

- **Over-escalation still fails.** When a triage case over-escalates (e.g. a vague ticket read as `critical`), the grader returns `pass: false` — every rubric's fail list makes over-escalation a hard fail, and the grader honors it. Guardrail 2 (never-override-a-fail asymmetry): a `pass: false` **always fails**, at any score. Honor-`pass` cannot touch it.
- **Malformed JSON still fails, grader-independent.** Moved to a deterministic `is-json` assertion (companion hardening §4), AND-ed with the rubric. It fails **outside** any grading mechanism — honor-`pass` never sees it.
- **Out-of-enum still fails, grader-independent.** Moved to a deterministic enum-membership check on `urgency`, AND-ed with the rubric. Same: fails outside the mechanism.
- **A genuinely-bad-but-`pass:true` output still fails.** Guardrail 1 (hard floor): if the grader emits `pass: true` with a score below the floor (0.6), the assertion still fails. Honor-`pass` operates only in [floor, 0.8), never below.
- **A rubber-stamping grader is still caught.** ADR-0007 §5.2's per-eval must-fail canary must FAIL under a correctly-wired harness; if a grader has degraded into stamping `pass: true` on everything, the canary passes → the gate **errors** (harness-broken), not silently green. §5.3 grader≠target keeps a model from grading its own blind spots. Both are already required checks; honor-`pass` leans on them, and they are exactly the controls designed for this.

Why it can't be gamed into a rubber stamp: to weaken the gate through this mechanism, an actor would need the grader to return `pass: true` on something that should fail — but over-escalation yields `pass: false` (asymmetry blocks it), contract violations are checked deterministically (the mechanism never sees them), a low-scored `pass: true` is floored out, and a grader that lies `pass: true` on everything trips the must-fail canary into a hard error. There is no path from "honor the grader's `pass` verdict, floored" to "pass a thing that should fail." The only loosening the mechanism introduces is confined to a rubric-sanctioned borderline band the grader itself forgave; every other change it makes is a *tightening* (it now honors a grader's explicit `pass: false` even at a high score, which today's raw `score >= 0.8` rule would have passed). **Met.**

### (c) Cost / latency budgeted — EXIT CRITERION

**The chosen mechanism (honor-`pass` + deterministic split) adds ZERO metered cost and ZERO added latency per gate run.** The grader's verdict — `score` *and* `pass` — is produced by the **single existing judge call** (FQ-77's `pass:true, score:0.75` transcript is the direct evidence both fields are already emitted). Honor-`pass` needs no extra call; it re-decides the pass/fail *locally* from that existing verdict. **Mechanically, the build cannot literally "read one more already-parsed field"** — the live `threshold: 0.8` decides the assertion by `score >= 0.8` and *discards* the grader's `pass` boolean (that discard is the bug), and the never-override-a-fail asymmetry (guardrail 2) is impossible to implement from `score` alone. So the build must **relocate the threshold decision out of the per-assertion config and re-decide (`grader.pass && score ≥ floor`) in the harness** (review Finding 1 / C1). That relocation is local compute on an already-returned verdict — it adds no LLM call. The deterministic assertions (`is-json`, enum regex/JS) run locally in the promptfoo process — no LLM call. No extra target calls, no extra judge calls; the ADR-0007 §4 cost model is **unchanged** (~$0.03/full-suite run, ~$5.50 CAD/month at 130 runs, well under the $10 CAD hard cap).

Contrast, quantified, the rejected default: **multi-sample N=3 (judge-only) ≈ triples the judge-dominated cost** — from ≈ $0.027 to ≈ $0.081 of judge spend per full-suite run (target stays ≈ $0.004), pushing the monthly from ≈ $5.50 CAD toward ≈ **$15 CAD at 130 runs/month, over the $10 CAD hard cap**, and adding **2× judge latency** to every prompt-touching PR. That overrun is precisely why multi-sample is not the default. **As an optional per-case escalation** it is bounded: applied to (say) 2–3 empirically-(P2)-prone cases at N=3, it adds ≈ 4–6 extra judge calls per full-suite run ≈ +$0.01/run ≈ +$1.50 CAD/month — a rounding error that stays comfortably under the cap. **Met.**

Per the Sprint 12 §5.1 standing norm (FQ-77's banked lesson, category (a)): **its merge will require the user's OWN direct, in-the-moment authorization. Standing self-merge authorization does NOT cover it, and a coordinator relay of that authorization does NOT satisfy it** — the governance classifier correctly refused both in FQ-77 (a relayed claim *and* a direct merge attempt built on it), until the user's own fresh, un-relayed words landed the change. Whoever executes the Sprint 14 build must **name this in the build task's exit criteria and plan to pause and ask** — not assume standing authorization carries.

Therefore `live-eval-gate` neutral-skips on this PR (it touches no prompt surface), and this PR is normal-self-mergeable under standing authorization: the boundary binds the *build*, not the *design doc*, and this document is careful to be exactly that distinction. **Named. ### 3. Sprint 14 build scope — sized, named, scheduled (exit criterion)

**Small-to-Medium. A focused mini-track, not a research project** — the mechanism is a change to how one existing decision (per-assertion pass/fail) is computed, plus a mechanical assertion rewrite, plus a re-baseline. Ordered so nothing enforces until a green baseline exists under the new logic:

1. **Relocate the threshold decision + implement honor-`pass` + guardrails in the shared runner (S–M) — review C1/C2.** In the `live-run.*` harness / grading step (ADR-0007 §6), **remove/neutralize the per-assertion `threshold: 0.8`** so promptfoo surfaces the grader's unmodified `{pass, score}`, and re-decide in the harness: `fail if score < floor; else pass if (grader.pass === true); else fail` — i.e. the collapsed rule `pass ⟺ grader.pass && score ≥ floor`, with the never-override-a-fail asymmetry (a grader `pass:false` is always a fail) explicit. **Define the write-point (C2):** stamp the final honor-`pass` verdict into the results-JSON `success` / `gradingResult.pass` *before* `compare-baseline.py` reads it (its `_entry_passed` reads `success` first), so the baseline comparator stays a pure file diff and the decision logic is not split across two files. Slightly larger than "read one field," still S–M.
2. **Split objective contract checks into deterministic assertions (S–M) — review C5.** For each product eval, add `is-json` + an objective-contract check (for triage: a closed-enum membership check on `urgency`; ticket-respond and kb-learn each have their *own* objective contract — confirm each before splitting, do not assume the triage enum) as deterministic assertions AND-ed with the existing `llm-rubric`, and trim the now-redundant objective clauses' *decision weight* from the rubric prose (keep the prose as guidance; the deterministic assertion is the enforcer). When the harness recomputes row `success` (step 1), it must **AND the native deterministic components with the honor-`pass`ed llm-rubric component** — honor-`pass` operates on the rubric component only and must never shadow a deterministic fail. Touches all three `evals/*.yaml`. This is a rewrite of existing assertions, **not** new coverage — scope-distinct from the ≥20/eval growth arc.
3. **Calibrate the floor and band (S).** Empirically set the floor (candidate 0.6) strictly below the observed borderline band (0.7–0.75) using the suite's real grader-score distribution across a handful of runs; identify which cases (if any) are genuinely (P2)-prone and warrant the optional `multiSample` opt-in. Document *why* each value was chosen (rationale matters when a threshold later looks arbitrary).
4. **Re-validate every canary fixture under the new mechanism (S) — review C4.** `calibration-guards.py::cmd_canary_check` requires the must-fail canary to have `success is False`; a must-fail canary that fails *only* because its score sits in [floor, 0.8) would now **pass** under honor-`pass` and wrongly trip the rubber-stamp guard into a hard error. Re-validate each canary so its must-fail case fails via a grader `pass:false` or a score below the floor — never via the old threshold gap — and apply the honor-`pass` stamping (step 1) to canary runs too, so the guard reads the same verdict the product tests are judged by.
5. **Re-capture the green baseline under the new mechanism (S) — review C3.** Per ADR-0007 §5.5: run all three evals green against `main` under honor-`pass` + deterministic split, record in `eval_gate_runs` (the existing `case_results` JSONB column already carries per-case detail — genuinely no new schema) + LangFuse. Re-capture is **load-bearing for two independent reasons**: the per-assertion result feeding the comparator changes, *and* previously-green ≥0.8 cases can flip to fail if a grader returns `pass:false` at high score (the bidirectional delta, §(b)). The baseline-relative gate (ADR-0007 §5.4b) then compares future runs against *this* baseline; `compare-baseline.py` fail-closes on any regression against it.
6. **Governance-gated merge (§(d)).** The change modifies the shared safety net → **pause and get the user's own direct authorization to merge** (Sprint 12 §5.1). Do not self-merge on green.

No new database schema (`eval_gate_runs` exists, T-58; honor-`pass` changes the *value* of the per-assertion result, not its storage shape). No new external service. No new credential or key/scope (the mechanism reads existing fields and runs local checks) — so the Sprint 9 §5.1 credential-review gate does **not** fire for the chosen option; it would only fire if the optional multi-sample escalation were later wired in a way that needed a separate judge-budget key, which the design does not require. The single load-bearing build judgement is the floor/band calibration (step 3); everything else is mechanical.

## Consequences

- **Positive:** the gate stops blocking correct changes for borderline nuance its own grader forgave — the exact FQ-77 failure (a proven security fix blocked for a reason the grader disagreed with) cannot recur. It fixes the *presenting* case (P1) that multi-sample would have missed, at **zero added cost/latency** (reads an already-returned field), **provider-neutrally** (structured fields + numeric config, no model assumption), and **without weakening the gate** (§(b): the delta is bidirectional and drop-don't-weaken-safe — it stops failing a borderline case the grader passed, *and* starts failing a high-score case the grader explicitly failed; over-escalation, malformed, and out-of-enum all still hard-fail, now some of them *more* robustly because they move to deterministic checks). It retires the per-case prose-idiom treadmill (Option 0's forever-loop of governance-gated one-line calibrations) in favor of one structural rule. The deterministic objective-assertion split is a real robustness gain in its own right: objective contract conformance stops being subject to grader variance at all.
- **Negative / accepted:** the gate now **trusts the grader's `pass` boolean** in the borderline band, leaning harder on the already-required canary + grader≠target guards (ADR-0007 §5.2/§5.3) — accepted, because those are exactly the controls built for this and a rubber-stamping grader is caught by the must-fail canary erroring the run. A genuine, honest consequence to log plainly: **sub-threshold-but-passing quality drift is now, by definition, within the rubric's stated tolerance** — if the *tolerance itself* is too loose (e.g. the suite should stop accepting `low` on vague tickets), that is a **rubric-authoring decision for the monthly coverage review**, not something the gate mechanism will catch, because the mechanism now faithfully honors what the rubric says. The hard floor bounds how far that can drift (nothing below 0.6 passes). This is the correct division of labor — the *mechanism* decides faithfully; the *rubric* decides what "faithful" means — but it does move a sliver of responsibility from the automatic gate to the human rubric-review cadence, and that is named, not hidden.
- **Explicitly out of scope:** eval-coverage growth toward ≥20/eval (a separate additive arc — different problem, different answer); all other open carries (T-107 staging re-pin, T-98 monitor threshold, provider-credential class). None are grader-robustness.
- **Open item that resolves at build time (Sprint 14):** the floor/band calibration (§3 step 3) is the single load-bearing build judgement; and the **build's merge is governance-gated (§(d))** — the user's own direct authorization, planned for and named in the build task, not assumed from standing self-merge.

## Evals Lead Design Rationale

- **Reviewer:** Evals Lead (Prompt Quality) — author of this ADR; this section is the design rationale that stands in for the usual appended review, per the ADR-0007/0008 convention (where the author *is* one of the deciders, the rationale is inline and the *other* decider's review is appended).
- **Date:** 2026-07-13
- **Verdict on scope:** this is an agent-owned technical design (CONSTITUTION "could a senior engineer answer this by reading the repo?" → yes). **No FOUNDER_QUEUE escalation is required by this design** — it neither relaxes a standing constraint (it *tightens* the objective checks and merely stops the gate mis-deciding borderline nuance) nor forces a provider switch with a quality delta. The one founder-facing touch is the eventual **build's** governance-gated merge (§(d)) — a Sprint 12 §5.1 authorization boundary, not a business decision.
- **Why honor-`pass` over multi-sample (the call I'd defend hardest):** multi-sample is the intuitive answer to "the grader is noisy," but it does not fix the case that actually blocked the security fix. Case (g)'s grader was not noisy — it was *honest and stable* at 0.75, deliberately docking a nuance its own rubric forgives while explicitly returning `pass: true`. Averaging honest 0.75s changes nothing. The real defect is that we **threw away the grader's `pass` boolean** in favor of a raw number the grader itself had already contextualized. Honoring the boolean is not a loosening — it is *reading the answer the grader gave us instead of overriding it with a threshold comparison it never intended.* That it also costs nothing and is provider-neutral is a bonus; the reason it's right is that it fixes P1.
- **Why the deterministic split is non-negotiable for constraint (b), not optional polish:** honor-`pass` alone is *canary-backstopped* against a rubber-stamping grader — good, but it still routes objective contract conformance (valid JSON, closed enum) through a subjective grader whose `pass` we now trust more. That is the one place the design could be gamed if a grader ever conflated "minor nuance" with "malformed output." Pulling the objective checks out into deterministic assertions closes that seam entirely: the mechanism *cannot* honor-away a malformed or out-of-enum output because it never grades those in the first place. It converts §(b) from "trust the grader + canary" to "the objective failures are grader-independent, full stop." That is the difference between an argument and a proof.
- **Why multi-sample is retained (not deleted):** (P2) variance is real — a respond case genuinely flapped in FQ-77 on a PR that didn't touch it. Honor-`pass` narrows the band that flapping matters in, but a case whose true mean sits *right at* the floor could still scatter across it. Multi-sample is the correct tool for those specific cases; the error is making it the *default tax* on every run. Keeping it as a per-case, calibration-gated opt-in puts the cost exactly where the problem is.
- **Handoff:** Tech Lead — the CI/harness-integration review below is yours; the load-bearing items are (1) whether the honor-`pass` + floor + asymmetry logic lands cleanly in the `live-run.*` grading step without disturbing the baseline-relative comparison (ADR-0007 §5.4b), and (2) whether the deterministic-assertion split is best expressed as `is-json` + a `javascript` enum check or some other promptfoo-native shape. Neither changes the grading-mechanism *design* I authored. PM — this ADR is the decision-of-record; the Sprint 14 build is sized in §3 with the §(d) governance gate folded into it. No founder escalation required by the ADR itself.

### Findings (each verified against the repo, not taken from the ADR's description)

1. **The load-bearing feasibility fact is REAL in its verified half and forces a build consequence the ADR's prose understated. — the single most important finding.** Two halves, treated separately as ADR-0008's PM review treated the Coolify-alias fact:
   - **(i) "The `threshold` discards the grader's `pass`" — VERIFIED, primary-source.** FQ-77 run `29215571461` (DECISIONS.md) records the grader returning `pass:true, score:0.75` with the verbatim reason that `low` is not a listed fail value, and the assertion nonetheless FAILING because `threshold: 0.8` decided it by `score >= 0.8`. `evals/ticket-triage.yaml` confirms every llm-rubric assertion carries `threshold: 0.8`. So the override direction the whole ADR rests on is not a hypothesis — it is observed behavior of promptfoo 0.121 as configured today.
   - **(ii) "The runner reads the `pass` field the grader *already returns*" (§(c), and §3 build-step-1 "the fields are already parsed") — MECHANICALLY INACCURATE as written, and this is unavoidable, not incidental.** The honest implementation is not "read one more field that's already in the JSON." It is: **remove/neutralize the per-assertion `threshold: 0.8` so promptfoo surfaces the grader's *unmodified* `{pass, score}`, and relocate the entire floor + honor-`pass` + asymmetry decision into the harness.** The decisive argument is guardrail 2 itself: the never-override-a-fail asymmetry ("a grader `pass:false` **always fails**, at any score") is *impossible to implement from `score` alone* — score cannot distinguish "grader said fail at 0.75" from "grader said pass at 0.75," which is the entire premise of honoring the verdict. Recovering that boolean — which both honor-`pass` and the asymmetry require — is exactly what the live `threshold` throws away (per (i)). Therefore the threshold must be relocated. This conclusion holds **regardless** of whether promptfoo happens to retain a raw pre-threshold `pass` in some incidental JSON field, so the build should *not* bet on scraping it — it should relocate the decision. **Not Decision-changing** (Option 3 is still zero *metered* cost, still provider-neutral, still fixes P1), but the builder must size step 1 as "relocate the threshold decision + recompute row `success`," not "read an already-parsed field." _(Author: incorporated — §(c) and §3-step-1 reworded; Status note carries the correction.)_

2. **The mechanism collapses to a single rule, and the behavioral delta is BIDIRECTIONAL — §(b)'s "strictly narrows" is imprecise (still drop-don't-weaken-safe).** Worked through, honor-`pass` + floor + asymmetry reduces to: **`pass ⟺ (grader.pass === true) AND (score ≥ floor)`.** The `0.8` threshold is *subsumed*, not a surviving boundary. Consequence the ADR's prose slightly misstates: in the `[0.8, 1.0]` region, a grader `pass:false` at score 0.85 **now fails** where today it passes on `score >= 0.8` alone. That is a *tightening*, so drop-don't-weaken (§(b)) still holds — but the delta is not "strictly narrows the set of blocking failures to exclude exactly one thing" (§(b)/Consequences); it removes one class of block (borderline nuance the grader forgave) **and adds another** (high-score outputs the grader explicitly failed). Both directions are correct behavior; the ADR should state the delta as bidirectional rather than purely subtractive. This *reinforces* the design — it is a second, independent reason the re-baseline (build step 4) is mandatory, not ceremonial (Finding 4). _(Author: incorporated — §(b) now states the collapsed rule and the bidirectional delta explicitly; Consequences updated.)_

3. **Where the honor-`pass` verdict must be STAMPED is the load-bearing integration detail the ADR's "feeds the baseline but needs no schema change" glosses.** `compare-baseline.py::_entry_passed` reads `entry["success"]` first (row-level), falling back to `gradingResult.pass`. Today `success` = post-threshold (`score >= 0.8`). Once the threshold is relocated (Finding 1ii), promptfoo's raw `success` reflects the grader's unmodified `pass` **with no floor applied** — which is *not* the honor-`pass` verdict (worked example: `pass:true`/score 0.4 → promptfoo raw `success=true`, but honor-`pass` verdict = **fail** via the floor). So the harness **must overwrite the results JSON's `success` / `gradingResult.pass` with the final honor-`pass` verdict *before* `compare-baseline.py` reads it**, keeping the comparator a pure file diff. The alternative — teaching the honor-`pass` rule into `_entry_passed` — splits the decision logic across two files and is not recommended. Name this write-point in the build. **Confirm the positive too:** "no new schema" is **genuinely true here** — the `case_results` JSONB column already exists (migration `20260709020000`, referenced in `compare-baseline.py`), and honor-`pass` changes the *value* of `passed` per case, not the shape. This is a clean contrast with ADR-0007's Finding 4, where I had to flag "no new schema" as conditional; here it is unconditional. Good. _(Author: incorporated — write-point named as C2 in §3-step-1.)_

4. **Re-baseline (build step 4) is mandatory for TWO independent reasons — keep it ordered first.** The ADR already orders it correctly (nothing enforces until a green baseline exists under the new logic), which is right. Beyond the ADR's stated reason (the per-assertion result feeding the comparison changes), Finding 2 adds a second: previously-green `≥0.8` cases can *flip to fail* if any grader returns `pass:false` at high score. Both cut the same way — the last green baseline captured under the old `score >= 0.8` rule is not valid under the new rule, and `compare-baseline.py` fail-closes on any regression against it (`baseline PASS -> current FAIL = BLOCK`). Re-capture is load-bearing, not hygiene. _(Author: incorporated — §3-step-5 now names both reasons.)_

5. **Canary re-validation is a real, concrete build condition.** `calibration-guards.py::cmd_canary_check` reads `row.success` and requires the must-fail canary to have `success is False`. A must-fail canary that fails *only* because its score sits in `[floor, 0.8)` (e.g. 0.75 < 0.8) would, under honor-`pass`, **now pass** and wrongly trip the rubber-stamp guard into a hard error (exit 4). The build must re-validate every canary fixture so its must-fail case fails *under the new mechanism* — i.e. via a grader `pass:false` or a score below the floor, never via the old threshold gap. (The must-pass canaries are unaffected; honor-`pass` only ever adds passes in the band or removes them below the floor.) This also means the honor-`pass` stamping (Finding 3) must be applied to the canary runs too, so `cmd_canary_check` reads the same verdict the product tests are judged by. _(Author: incorporated — §3-step-4 is the canary re-validation step.)_

6. **The deterministic-split companion is the right promptfoo-native shape, with two caveats.** `is-json` (built-in) + a `javascript` assertion parsing the output and checking `urgency ∈ {critical,high,normal,low}` is correct: promptfoo AND-s sibling assertions into row `success` by default, so a malformed-JSON or out-of-enum row hard-fails *outside* the llm-rubric component regardless of the grader's verdict — which is exactly what makes §(b) a proof rather than an argument. Two build caveats: **(a)** once the threshold is relocated (Finding 1), the harness recomputes row `success` and must **AND the native deterministic components with the honor-`pass`ed llm-rubric component** — the honor-`pass` logic operates on the rubric component only and must not shadow a deterministic fail. **(b)** The enum check is **triage-specific** — `evals/ticket-triage.yaml` has the closed `urgency` enum; ticket-respond and kb-learn have their *own* objective contracts. "Touches all three evals" (§4 / build-step-2) is a per-eval rewrite of *each eval's own* objective clauses, not a uniform enum. A one-line "confirm each eval's own objective contract before splitting" suffices; no need to expand scope. _(Author: incorporated — §3-step-2 carries both caveats as C5.)_

7. **Cost/latency claim (§(c)) — CONCUR, zero added metered cost, verified against the harness.** Honor-`pass` reads `score`/`pass` already produced by the single existing grader call — relocating the threshold decision adds no LLM call, only trivial local compute. The deterministic assertions (`is-json`, `javascript`) run in-process — no model. The multi-sample contrast holds arithmetically against ADR-0007 §4: N=3 judge-only re-grades ≈ 3×$0.027 ≈ $0.081 judge + $0.004 target ≈ $0.085/full-suite run × 130 runs/mo ≈ $11 USD ≈ ~$15 CAD, over the $10 CAD hard cap — and the ADR correctly scopes multi-sample as re-grading a *single* target output (target stays ~$0.004), so the math is sound. Retaining multi-sample as a bounded per-case escalation is the honest home for it. **Concur.**

8. **Scope discipline — CONFIRMED sound.** Excluding the ≥20-tests-per-eval coverage-growth arc is correct: it is orthogonal. `compare-baseline.py` already implements the baseline-relative gate (ADR-0007 §5.4b) precisely so N=4 does not force an absolute 100%; grader-robustness is about *how one borderline assertion is decided*, coverage-depth is about *how many assertions exist*. They are different answers to different questions, as the ADR states. Folding them would repeat exactly the overcommit pattern Sprints 6–12 held against.
