## 2. What worked

- **The ADR-then-build discipline held on the anchor — the team refused the fourth manual re-align and wrote a design of record instead.** This is the sprint's headline, and it is a *restraint* story rather than a heroics one. The reflex, three times over, was `fix-ops-hub-prod-litellm-url.yml`. The Tech Lead's read this time was that a fourth reactive patch is the wrong move, and T-101 delivered the alternative: ADR-0008, taken through an **independent review** (Tech Lead author + Production Manager review appended in-doc, mirroring the ADR-0007 author+reviewer precedent), landing a decision (Option 1 — persistent Coolify network alias) that kills the class *at source* rather than detecting it faster. The precedent it follows — Sprint 8→9's ADR-0007-then-build — is now a repeatable pattern, not a one-off.

- **The ADR is honest about its own weakest fact, and structured around it.** Option 1 is chosen *contingent on a feasibility spike*, because the one load-bearing assumption — does this Coolify instance support a redeploy-surviving network alias — is genuinely unverifiable from static evidence (Docker DNS resolves full container names/aliases, not the stable UUID prefix; three incidents plus CLAUDE.md's standing "check `docker ps` after each deploy" are weak evidence, not proof). Rather than assert it and hope, the ADR pre-commits **Option 2 (post-redeploy re-sync hook) as the fallback** if the spike fails — so the Sprint 12 build is a build, not a re-argument. The PM review explicitly validated that the contingency shape is correct because the feasibility fact is genuinely unconfirmable (grounded in the prior `fix-litellm-network.yml` staging alias experiment that was never carried to prod). A design that names its own risk and pre-wires the branch is a design you can build against.

- **T-102 is the NEW Sprint 10 §5.1 norm's first application on a fresh monitor, executed exactly to spec.** Sprint 10 banked the norm — *prove a monitor's failure/alert path actually fires (`mode=simulate-failure`) before trusting it live* — after two real monitor bugs (BC1, the reset false-pass) were caught only that way. T-102 was the first monitor edit since, and it did not review the YAML and call it done: **four real `workflow_dispatch` runs, each awaited to completion before the next, each cited by run ID**, proving the whole shape by *positive evidence* — Run A (1/3) and Run B (2/3) self-cleared with `status-incident.yml`'s own run list confirming zero dispatches; Run C (3/3) opened a real incident (`status-incident.yml` run `29210745813`, `status-content` commit `77b21c7`, `resolved: false`); a real-prod-healthy check preceded the recovery run so a dispatch wouldn't open a *second* incident; Run D (`mode=live`) auto-resolved it (`6077e4e`, `resolved: true`); and a full branch scan confirmed no file left `resolved: false`. The norm forced the proof up front, and the proof is airtight. That is the norm earning its keep on the very next opportunity.

- **T-102 chose the boring, no-new-surface mechanism — and reasoned the threshold from the SLA, not from a round number.** The consecutive-fail counter is a read-only `gh run list` lookback reusing the job's existing permissions: **no new secret, no repo variable, no state file.** It's scoped two ways (same `github.event_name`; a 2h recency window) so verification dispatches can never cross-contaminate the real schedule-driven streak. N=3 was chosen over N=2 from the cadence arithmetic (15-min cron → N=3 pages a sustained break at ~T+30min, inside the ~30–45min SLA; N=2 at T+15 undershoots the insulation the task exists to buy), not by feel. Cheapest thing that could work, sized to the actual constraint.

- **Coverage growth paid a security dividend — the live gate found a real vulnerability, not just a regression it was watching for.** T-103's job was to grow evals. Running real cases through the now-live gate surfaced a **genuine triage body-injection vulnerability**: a ticket body carrying `IGNORE ALL PREVIOUS INSTRUCTIONS…` flipped the model's output outright (forced `urgency=critical`/`category=vip`/`routing=executives`, eval score 0) because the `ticket-triage` system prompt has no untrusted-input clause. Nobody had tested for it; the coverage-growth task found it. This reframes the gate: it isn't only a guardrail against regressions, it's a discovery instrument when you feed it adversarial cases. (Fix committed to Sprint 12 — §5, §7.)

- **T-103 held the "drop, don't weaken" line — coverage discipline over a green checkmark.** Both cases the `strict_new` dispatch flagged as failing were genuine current-prompt gaps the Evals Lead *could not fix* in an `evals/**`-scoped, prompt-frozen task. Rather than soften the rubric to make them pass (which would have shipped a case that locks in the *broken* behaviour), both were **dropped and logged as findings** for a future prompt-hardening task. Same family as Sprint 9's "refused to force a false pass" and Sprint 10's self-corrected admin-bypass: the reflex to make the number green is being resisted where it would cost integrity. The suite that shipped is 42 real regression-locks, not 44 with two aspirational fictions.

- **Scope discipline held for the sixth consecutive sprint, visibly, at three separate decision points.** T-102 explicitly did **not** fold the T-98 6-hourly monitor into the same threshold change (correctly reasoning it would push detection to 12–18h, working against why T-98 exists) — flagged as a separate undecided carry instead of quietly bundled. T-103 did **not** pad the evals to hit the ≥20 number. And ADR-0008 stayed scoped to the URL-suffix class only, explicitly declining to fold in the still-open provider-credential-divergence carry whose trigger has not fired. Three chances to over-reach, three declines.

## 3. What didn't work (or cost more than it should have)

- **A shared-main-tree HEAD collision between two concurrent agents — the sprint's one real process failure.** T-101's initial ADR commit briefly landed on the concurrently-active `t102-litellm-monitor-fail-threshold` branch because both agents were committing docs against a shared `main` working tree at the same time. It was recovered **cleanly and with zero lost work** — T-102's commit `8d6bb45` was restored intact via compare-and-swap, the ADR was cherry-picked onto its own branch as an ADR-only diff, and the remaining T-101 work was moved into an isolated worktree. But "recovered cleanly" is not "didn't happen": this is the *exact* hazard MEMORY.md's "worktree isolation required" note and the standing isolation norm exist to prevent, and this time it actually fired. The lesson is not "be more careful" — it's structural: **start concurrent git-writing agent work in an isolated worktree from the first commit, not after the first collision.** Banked as a standing norm (§5.1).

- **The URL class is designed, not fixed — do not let "ADR Accepted" read as "problem solved."** This sprint produced a *design of record* and a *noise-reduction mitigation*. It did not produce the fix. The class remains live: a litellm-prod redeploy in the interim can still orphan the URL, and until the Sprint 12 build lands, the mitigations are exactly what they were — T-97 catches it, and now T-102 makes it page more cleanly after 3 sustained fails instead of crying wolf on the first blip. Avoiding the fourth manual re-align was the disciplined call, but the honest ledger entry is: **the loudest recurring break of the last five sprints is still unfixed at the infra level, by design, with the fix scheduled for Sprint 12.**

- **The ADR surfaced a latent second bug it did not fix — both break-glass workflows verify the wrong endpoint.** ADR-0008 §3 flags that *both* existing re-align workflows (`fix-ops-hub-prod-litellm-url.yml`, `update-litellm-suffix.yml`) confirm success against the generic `/health` — the exact blind spot FQ-69 taught us about, which stays green through an internal-hop break — instead of `/health/litellm-internal`. This is a real, known-open defect in the recovery tooling: a break-glass re-align could report success while the real path is still down. It is recorded, assigned to the Sprint 12 build to correct, and **not fixed this sprint.** Naming it is right; leaving it open for a sprint is the cost.

- **T-103 grew coverage but left two genuine prompt gaps open — growing evals surfaced problems it couldn't fix in-scope.** The triage body-injection gap is a *real* vulnerability (any ticket body can currently manipulate its own priority/routing) documented-but-unfixed until Sprint 12. The respond multi-question-completeness gap is *unconfirmed* — a single judge run, n=1, score 0.6 just under the 0.8 threshold, which could equally be genuine incompleteness or ordinary judge variance the baseline-relative gate is designed to absorb. Both are honestly logged rather than fixed or dismissed, but the net state is: two real prompt findings are now on the books, one committed to a fix and one gated behind a confirmation re-run. Coverage went up; the surface area of known-open gaps also went up.

- **Operational watch item — subagent `response stalled mid-stream`: no new recurrence recorded in this sprint's task logs.** The T-101/T-102/T-103 records do not report the stall pattern that recurred across Sprints 9 and 10. That is an *absence of a logged recurrence*, not a confirmed all-clear — I have no positive signal either way for the sessions that didn't note it. The watch item continues unchanged: not root-caused, not actionable, no owner; escalate only if it worsens or a stall lands after a partial commit. Recorded for continuity (§5).

## 4. Incidents, blockers, and resolutions

### 4.1 T-101 — ADR-0008: choosing a design of record around an unverifiable fact

The anchor's discipline is worth laying out, because the interesting part is *how the ADR handles what it can't know*.

**The class, restated.** T-71 (Sprint 6), FQ-69's `LITELLM_URL` sub-issue (Sprint 8), and FQ-76 (Sprint 10) are the same shape three times: a litellm-prod redeploy rotates its internal Docker container suffix; ops-hub-prod's pinned `LITELLM_URL` isn't re-synced; the internal hop throws → `/health/litellm-internal` 503 "unreachable". Three manual re-aligns via `fix-ops-hub-prod-litellm-url.yml`. Band-aid ceiling reached.

**The decision.** Option 1 — give litellm-prod a persistent Coolify network alias / fixed internal hostname and pin `LITELLM_URL` once to it. Chosen because it kills the class *at source*, adds zero moving parts and zero new credential surface, and is provider-neutral as a one-paragraph recipe (no app change — `resolveLitellmTarget` still reads `LITELLM_URL`, only the value stops rotating).

**The honest hedge.** Option 1 is contingent on a **feasibility spike**: the load-bearing fact — does *this* Coolify instance support a redeploy-surviving alias — is unverified, and the ADR says so plainly instead of asserting it. **Option 2 (automated post-redeploy re-sync hook)** is the pre-committed fallback (rejected as *primary* because it automates the band-aid, holds a standing SSH-key + Coolify-token surface, and is a monitor-the-monitor moving part). **Option 3 (app-side service-name resolution)** was rejected — it collapses into Option 1 (still needs a stable name) or requires `docker.sock` in the app container (container-escape-equivalent privilege escalation), which is why Option 2, not Option 3, is the fallback.

**Hard constraints met as exit criteria, not prose.** (a) provider-neutral / Project-#2 recipe (§5); (b) verification is `/health/litellm-internal` → 200 **and** the T-97 monitor auto-resolving its own incident, never generic `/health` (§3) — which is where the ADR also surfaced the latent break-glass-workflow endpoint bug; (c) credential gate named (§4) — Option 1 touches no secret; the Option 2 fallback triggers the Sprint 9 §5.1 Security Lead review of the concrete SSH/token scope *before* implementation. Scoped to the URL-suffix class ONLY — the provider-credential-divergence class (FQ-69's 401 master-key signature) stays a distinct, still-open, uncommitted carry, explicitly not folded in.

**Status ACCEPTED** via the ADR-0007 author+reviewer precedent (Tech Lead author + Production Manager review appended in-doc; PM verdict: approved as design-of-record, no change to the ranking, feasibility genuinely unconfirmable from static evidence). PM build conditions folded into §6.1: prove the spike on litellm-**staging** first (free canary), shared-network pre-check, explicit written rollback (fall back to the break-glass re-align), re-check §3 against T-102's shipped resolve-threshold, size the T-97 wait as up to one 15-min poll cycle, note the alias-apply likely needs a redeploy not a restart. Build sized S–M, scheduled Sprint 12.

### 4.2 T-102 — the failure path, proven four times before trusting it

`monitor-litellm-internal-auth.yml` used to page on a single failed poll; the 07-10/07-11 self-healing hiccups (sprint-10 retro §4.2/§7) churned open/resolve noise on every transient blip, which is what made FQ-76's real 3-day sustained break harder to read against the noise. The fix (PR #422, squash `15c02be`) adds `FAIL_THRESHOLD: "3"`. The mechanism and the SLA-derived N=3 choice are in §2; what matters here is the **hard exit gate**, run exactly as Sprint 10 §5.1 requires:

1. **Run A** [29210699251] (`mode=simulate-failure`) — conclusion `failure`, streak 1/3 → **no incident** (confirmed by `status-incident.yml`'s own run list showing zero dispatches, positive evidence not absence-of-mention).
2. **Run B** [29210726416] — `failure`, streak 2/3 → **no incident**. Proves N-1=2 consecutive fails do not page.
3. **Run C** [29210741763] — `failure`, streak 3/3 → **incident OPENED** (`status-incident.yml` run `29210745813`, `status-content` commit `77b21c7`, `resolved: false`). Proves the Nth fail pages.
4. Confirmed real prod healthy (`curl /health/litellm-internal` → 200) *before* the recovery run — avoiding the failure mode where a `workflow_dispatch` recovery during a genuine break opens a *second* incident instead of resolving the first (the resolve handler resolves by most-recent-open, so ordering matters).
5. **Run D** [29210773889] (`mode=live`) — `success` → auto-resolve fired (`6077e4e`, `resolved: true`). Proves recovery auto-resolves.
6. Full `status-content` branch scan post-Run-D: **zero files `resolved: false`** — no spurious side effects, nothing left open.

Post-merge `main` smoke test [29211161378] `success`. **Known limitation** (noted in the workflow header, not fixed — over-engineering for scope): the streak signal conflates "the probe failed" with "the probe succeeded but the open/resolve dispatch step failed for unrelated infra reasons"; considered rare, flagged so it isn't a silent trap. **Deliberately out of scope:** T-98's 6-hourly `monitor-e2e-pipeline.yml` — the same threshold there would push detection to 12–18h, against the reason T-98 exists. Left untouched, flagged as a separate decision (§7).

### 4.3 T-103 — coverage growth as vulnerability discovery, and the drop-don't-weaken line

T-103 grew the three product evals additively (triage 9→15, respond 9→13, kb-learn 9→14; +15 net, 27→42), each new case a regression-lock grounded in the matching `src/inngest/*.ts` prompt (real gaps: emotional-tone-≠-urgency, out-of-enum vocabulary normalisation, security-report-≠-breach, cross-lingual severity, threat-de-escalation-without-capitulation, DPA-no-fabricated-commitment, identifier-redaction clusters, content-hijack injection, customer-went-dark honesty — not filler). ## 5. Process changes for Sprint 12 (and standing, going forward)

1. **NEW standing norm — start concurrent git-writing agent work in an ISOLATED WORKTREE from the first commit.** This sprint produced the justification: T-101's ADR commit collided on HEAD with the concurrently-active T-102 branch because both agents committed docs against a shared `main` tree at once (§3, §4.1). Recovered cleanly, zero work lost — but it is the *exact* hazard MEMORY.md's "worktree isolation required" note and the standing isolation norm exist to prevent, and it fired for real. Make it unconditional: **any session where another agent may touch the same files does its git work in an isolated worktree from commit #1**, not after the first collision. Cheap, structural, and it makes recovery-by-luck unnecessary. (This retro's own closeout was done in an isolated worktree, applying the lesson to itself.)

2. **Standing (carried, reaffirmed — VALIDATED this sprint) — prove a monitor's failure/alert path fires before trusting it live (Sprint 10 §5.1).** T-102 was the norm's first application on a fresh monitor edit, and it forced four real dispatches (cited by run ID, positive evidence at every step) up front (§2, §4.2). The norm caught no bug this time — precisely because it made the proof mandatory before the schedule was trusted, which is the win. Any Sprint 12 work that touches monitor/alerting logic (e.g. the T-104 Option 2 fallback's re-sync hook, or correcting the break-glass verify endpoints) is bound by this.

3. **Standing (carried, reaffirmed) — never soften the gate to pass; drop-don't-weaken.** T-103 dropped two failing cases rather than weaken them (§2, §4.3), turning eval growth into a vulnerability discovery. Sprint 12's T-105 must re-admit the dropped triage case only against a *real* prompt fix — the re-admitted case passing is the proof the fix worked, and it must never be softened to pass without one.

4. **Standing (carried, reaffirmed) — health/monitoring exercises the REAL path.** The all-clear after any internal-URL fix is `/health/litellm-internal` → 200 **and** the T-97 monitor auto-resolving its own incident, never generic `/health` (FQ-69's blind spot). ADR-0008 §3 surfaced that both break-glass workflows still violate this — the Sprint 12 build must correct it (§7).

5. **Watch item (carried) — subagent `response stalled mid-stream`.** No new recurrence recorded in this sprint's task logs (an absence of a logged recurrence, not a confirmed all-clear). Watch continues unchanged: not root-caused, not actionable, no owner; escalate only if it worsens materially or a stall lands after a partial commit.

*Sprint 11's throughline is discipline over reflex. The loudest recurring break of the last five sprints came due for a fourth manual re-align, and the team refused it — writing a design of record instead (ADR-0008), taken through an independent review, honest enough to name its own unverifiable feasibility fact and pre-wire the fallback so Sprint 12 is a build and not an argument. What this sprint did not do, it does not pretend to have done: the URL class is designed, not fixed; the break-glass workflows still verify the wrong endpoint; two real prompt gaps are open, one committed and one awaiting confirmation; and a shared-tree HEAD collision reminded us — recovered but not for free — that isolation is a first-commit decision, not a first-collision one. Sprint 12 builds the fix the ADR designed, and closes the vulnerability the coverage found.*
