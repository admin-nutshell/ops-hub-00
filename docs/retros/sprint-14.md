## 2. What worked

- **T-109 was handed off honest-red, and the team held the line rather than re-running for a lucky green.** The pre-merge `live-eval-gate` run failed. The task correctly diagnosed *why*: the honor-pass mechanism itself had already gone green on two earlier runs against the same threshold-era baseline (41/42 stable), and the failure traced to a specific, already-logged flaky case unrelated to the build. The discipline that matters here is not diagnosing the flake — it's that nobody re-ran the gate hoping the flake wouldn't fire, which would have been a quiet form of gaming the exact gate this build exists to make trustworthy. Handing off red with a clear, evidenced explanation and letting the user decide is the correct behavior under a required-check-blocks-a-correct-change situation — the same shape as FQ-77, resolved the right way this time without a governance detour.

- **The re-baseline ordering constraint (C3/C4) was correctly treated as load-bearing, not a formality — and the correction that got it there was banked, not silently applied.** An earlier pass on this task nearly dispatched `capture-eval-baseline.yml` against the feature branch; the corrected understanding (recorded directly in the task's own exit criteria) is that a branch-scoped capture would have become the *global* baseline for every prompt-touching PR while `main` still ran threshold-era logic — poisoning every other in-flight PR and prematurely activating honor-pass repo-wide. The fix was procedural, not code: merge first (with the user's own authorization), recapture on `main` immediately after, as the very first post-merge action. This ordering was followed exactly — the recapture run landed 16 seconds after the merge commit.

- **The recapture's actual content was checked this closeout, not assumed from the green checkmark — and that check is what surfaced the sprint's one real finding.** "The workflow succeeded" and "the baseline is trustworthy" are different claims. Reading the run log directly showed all 43 cases passed with 0 rows restamped by honor-pass (the mechanism agreed with the old verdict on every row, informative on its own — it means the delta only bites on genuinely borderline runs, consistent with the design intent) — and separately showed the specific case flagged as flaky in Sprint 13/14's own records happened to pass cleanly on this particular run. Verifying primary-source data instead of trusting a pass/fail summary is exactly the discipline Sprint 13's retro (§2) praised in the ADR review; it repeated here at the closeout layer.

- **T-108's fix was verified against its own success condition, not just deployed and assumed.** The task's whole point was "workflow-file-only merges should no longer trip a staging rebuild." Rather than take that on faith, this closeout checked: did `main-deploy.yml` run against the T-108 merge commit itself (a workflow-file-only change under the OLD rules, i.e. the exact failure mode T-107 hit)? It did not. That is a real, falsifiable confirmation that the fix works, using the merge event as its own test case for free.

- **The carried live finding (staging running from Sprint 13's incident) was closed cleanly, in the same window as the fix that made it safe to close.** A dedicated stop-and-confirm run polled `docker ps` to empty and logged "SC7 restored" five seconds after T-108 merged — closing a loose end from Sprint 13's incident without it lingering as an unresolved carry into a third sprint.

- **Both shared-safety-net and prod-adjacent merges went through the user's own hand — the norm Sprint 12–13 built worked as designed, with no detour this time.** `gh pr view --json mergedBy` on both PRs confirms `admin-nutshell` (`is_bot: false`), not a bot account and not an agent-initiated `gh pr merge`. T-109 is category (a) (shared safety net); T-108 is category (b) (prod-adjacent). Both required the user's own direct authorization per the standing §5.1 norm, and both got it. This is the norm's success case, worth recording explicitly rather than only ever writing up the times it was tested by a violation.

- **Scope discipline held for the ninth consecutive sprint.** Anchor + one parallel track, neither track's scope crept (T-109 stayed inside ADR-0009's six conditions, explicitly declining the ≥20/eval coverage arc named as a different question; T-108 stayed inside its own decision (c) scope, declining to fold in the T-98 threshold decision it flagged as separate).

## 3. What didn't work (or cost more than it should have)

- **WORK.md went stale immediately after both merges and stayed stale until this closeout — the exact lesson Sprint 13's retro (§3) already named and asked to not repeat.** Sprint 13's retro wrote: "when the user accepts an outcome, update the live board in the same session, not one closeout later." Both T-109 and T-108 merged (18:25Z and 17:10Z respectively) with WORK.md still reading "PR OPEN, awaiting the user's own direct merge authorization... NOT merged" for both — language that was actively wrong, not just outdated, for however long elapsed until this session reconciled it against GitHub's actual state. No harm occurred (nothing depended on the stale text in the interim), but it is the second sprint running this exact hygiene gap has been named. It needs to stop being a lesson and start being a checklist step: the merge event itself should trigger the WORK.md update, in the same session/PR, not wait for the next planning pass.

- **The baseline-capture ordering discipline (merge-first, recapture-immediately) was followed correctly, but nobody checked whether the captured baseline was itself trustworthy before calling the task done.** This is the sprint's one real finding, and it's worth stating plainly: doing the *procedurally correct* thing (recapture post-merge, not pre-merge) is necessary but not sufficient. A recapture that happens to land during an unlucky run of a known-flaky case would have silently frozen a *failing* state as the new "expected" baseline too — either direction is a coin-flip outcome nobody was watching for. The fix isn't procedural this time; it's substantive: a case already flagged as non-deterministic should not be trusted from a single capture run in either direction without the mitigation ADR-0009 itself already designed for exactly this. That mitigation didn't ship this sprint because the finding that justifies it only surfaced at this closeout, one sprint after the mechanism that would need it landed.

- **The flaky case was known before this sprint started (logged as an out-of-T-109-scope follow-up finding) and its interaction with the FIRST baseline capture under the new mechanism was not scoped as a checked risk in T-109's own exit criteria.** Nobody was negligent — C1–C6 do not obviously imply "also audit case-level baseline luck" — but the flaky case and the re-baseline-first requirement were both already on the table simultaneously, and the connection between them (a lucky/unlucky single capture determines whether a known-flaky case reads as future-green or future-red) wasn't drawn until this retro. Worth naming so the next mechanism-build task treats "does the first baseline capture interact with any already-known flaky case" as a standing pre-close question, not a discovery.

## 4. Incidents, blockers, and resolutions

### 4.1 T-109's post-merge baseline froze a known-flaky case on a lucky run — the sprint's one real finding

**The mechanism itself is sound and proven.** All six of ADR-0009's conditions (C1: relocate the `threshold:0.8` decision into the shared runner; C2: floor = 0.6, provider-neutral; C3: stamp the verdict before `compare-baseline.py` reads it; C4: canaries re-validated, must-fail still fails via the grader's own sentinel; C5: per-eval objective-contract split; C6: each case's escalation allowed-set encoded deterministically) were implemented and independently proven — 17 Python unit tests, node deterministic tests (triage 67 checks, kb 7), and a full end-to-end dry run showing FQ-77's originally-blocked borderline case now correctly passes while a below-floor, an over-escalation, and a grader-`pass:false`-at-0.85 case all still correctly hard-fail. This is not in question.

**The gap is in what the first post-merge baseline capture froze, and nobody checked until this retro.** `capture-eval-baseline.yml` ran 16 seconds after the merge (run 29274409705, 18:25:43Z) and reported 43/43 green — every case in every suite passed, zero rows needed honor-pass to differ from the old threshold-era verdict. Read case-by-case, one of those 43 rows is the triage case testing a Spanish-language, single-user login ticket — already documented (DECISIONS.md, T-109's own follow-up finding) as **non-deterministically over-escalating to `high` at `temperature: 0`** on some runs, a pre-existing model-layer flakiness unrelated to and out of scope for T-109. On this particular capture run, the model happened to classify it correctly (`normal`, grader `pass: true`, `score: 1`) — a coin flip that came up green.

**Why this matters going forward.** Baseline-relative gating means "did this PR regress a case versus its last known-good state." A case whose *true* behavior flips between compliant and non-compliant from run to run, frozen into the baseline in its compliant state, will read as "PR broke it" the next time it happens to flip — on any prompt-touching PR, including ones that touched nothing related to triage classification. This is a real behavioral flip (the case's own deterministic C6 escalation check, strict `{normal, low}`, would independently fail an over-escalated run — it is not a grader-noise artifact), but it will still land as a false-attributed regression on an innocent PR, which is exactly the *experience* FQ-77 was about even though the underlying mechanism is different and sound.

**Not fixed here — deliberately, and this is the correct call.** ADR-0009 already designed the contingency for precisely this shape of evidence: retain optional per-case multi-sample grading, "wired only for cases empirically shown to be (P2)-variance-prone during the C3 calibration." This IS that empirical evidence — the first calibration run under the new mechanism. Wiring it now, mid-closeout, would be scope creep onto a task already merged and correctly closed. It becomes Sprint 15's anchor (T-110) instead: narrowly scoped to the one flagged case, not a blanket multi-sample tax (ADR-0009's own cost-cap rejection of that as the default still holds).

### 4.2 T-108's SC7 fix verified live against its own success condition

T-108's core claim was falsifiable: after the trigger fix, a workflow-file-only merge to `main` should no longer trip a staging rebuild+start. The merge commit itself (`d07c623`, changing only `.github/workflows/main-deploy.yml`) is exactly that test case, and it was checked — no `main-deploy.yml` run correlates to that commit. Separately, the live finding carried from Sprint 13 (staging left running by an earlier trigger-gap trip, independent of this PR) was closed in the same window: a dedicated stop-and-confirm run polled `docker ps` to empty and logged "SC7 restored," five seconds after the T-108 merge landed.

SC7 option (ii) — the monitor itself asserting or tolerating staging's own state, which T-108's own review named as the necessary backstop for a residual timing race option (i) alone cannot close (staging's 5-minute `sweepNewTickets` cron sweeps are live during the transient start→stop window) — was correctly left unimplemented rather than folded in under time pressure. It carries to Sprint 15 as the parallel track (T-111), with the standing Sprint 10 §5.1 norm (prove any monitor/alerting change on its failure path first) explicitly binding.

## 5. Process changes for Sprint 15 (and standing, going forward)

1. **NEW — update WORK.md's task row to reflect a merge in the SAME session/action that performs or observes the merge, not at the next planning/closeout pass.** This is the second sprint (13, now 14) this exact staleness has been named. The fix is mechanical: whichever agent action merges a PR (or is the first to notice a merge happened) updates that task's WORK.md row before ending the session, even if full sprint closeout waits for later. Treat "merged" as a state that must be reflected within the same working session it's observed, not batched into the next retro.

2. **NEW — a mechanism build's FIRST post-merge baseline/calibration capture gets its content spot-checked against any already-known flaky/variance-prone case before the task is called fully closed, not just checked for a green summary.** T-109 did everything procedurally correct (merge-first, recapture-immediately) but nobody cross-referenced the capture's per-case results against the already-logged flaky case until this retro. Going forward: if a task's own records already name a specific flaky/variance-prone test case, the first baseline capture under any new grading mechanism explicitly checks what that capture froze for that case, and flags it (as this retro does) if the freeze happened to land on a lucky/unlucky single run.

3. **Standing (carried, reaffirmed — VALIDATED this sprint, no detour) — self-merge authorization does not cross a shared-safety-net (cat. a) or prod-adjacent (cat. b) boundary; both require the user's own direct words.** Both T-109 and T-108 merged via the user's own hand (`mergedBy` verified, not a bot, not an agent `gh pr merge`) — the norm's success case, not just its tested-under-violation case (Sprint 13). Worth recording the clean case explicitly so the norm isn't only ever discussed in the context of an incident.

4. **Standing (carried, reaffirmed) — never soften the gate to pass; drop-don't-weaken, and keep any calibration in its own PR.** ADR-0009's guardrails (floor, never-override-a-fail asymmetry, deterministic C6 escalation split) all proved live in T-109's build and its dry-run tests. Sprint 15's T-110 must preserve this: multi-sample escalation is a *detection-stability* refinement for one named case, never a blanket loosening, and the deterministic escalation check for that case does not change.

5. **Standing (carried) — any monitor/alerting workflow change is proven on its FAILURE path before going live** (Sprint 10 §5.1). T-111 (Sprint 15) is a monitor-logic change by definition; this is its core exit criterion, not background.

6. **Standing (carried) — verify a design's load-bearing fact against the repo/logs before banking a task as done, not against a green checkmark alone.** This closeout's own method (reading the actual post-merge run logs rather than trusting "success") is what surfaced §4.1's finding. Apply the same standard going into Sprint 15's T-110/T-111 closeouts.

7. **Standing (carried) — start concurrent git-writing agent work in an ISOLATED WORKTREE from commit #1** (Sprint 11 §5.1). Held clean through Sprint 14; applies to T-110/T-111 if run in parallel.

8. **Watch item (carried) — subagent `response stalled mid-stream`.** No new recurrence logged in Sprint 14's task records. Watch continues unchanged; escalate only if it worsens or a stall lands post-partial-commit.
