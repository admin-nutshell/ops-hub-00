## 2. What worked

- **A wrong mechanism was caught before a single line of build code was written.** The sprint's own initial scoping (drafted at Sprint 14's closeout) assumed ADR-0009's multi-sample contingency was the fit for case (i)'s flakiness. Re-reading ADR-0009's actual cost model closely — it prices multi-sample as re-grading *one fixed target output* N times — against the raw evidence (the target model's own classification varying ~50/50 across runs) showed a categorical mismatch: multi-sample smooths grader noise on a stable output; this problem was the opposite, a stable grader judging an unstable output. This was corrected in its own small, labeled DECISIONS.md/WORK.md entry before either build agent was dispatched — the exact "record a miss as an addendum, not a silent patch" discipline this project has praised in every retro since Sprint 13. Cheap to catch early; expensive if it had shipped as a build task and only failed during implementation.

- **The Evals Lead's actual diagnosis vindicated the correction and went further than "not multi-sample."** Rather than accept "fixture ambiguity" as a given, the diagnosis explained *why* the ticket was ambiguous (it straddled the triage prompt's own literal high/normal boundary definitions) and cross-checked the model's own over-escalating output against an independent grader's judgment on that specific output — confirming `high` was a defensible-sounding but ultimately wrong read, not an alternate correct interpretation the gate was wrongly rejecting. That distinction is what separates "the fixture needs fixing" from "the gate is being too strict" — getting it backwards would have been a real drop-don't-weaken violation.

- **A tempting shortcut was considered and rejected on its own record, not silently avoided.** The Evals Lead's own build notes show it seriously considered widening the deterministic allowed-set to accept `high`, then rejected it for two independent reasons (it wouldn't even fix the flakiness, since honor-pass never overrides a grader's own fail; and it would delete the only test in the suite guarding that a single-user login problem stays out of `high`). Recording a considered-and-rejected shortcut, not just the path taken, is exactly the kind of transparency this project's culture rewards.

- **T-111's failure-path proof was real, not reasoned-through.** Rather than assert "this would skip cleanly if staging were running," the task extended the monitor's existing `mode` input with a new `simulate-staging-running` value and dispatched it for real. The coordinator independently re-verified the actual run — reading each step's conclusion directly via the GitHub API, not trusting the agent's narrative — and confirmed every downstream step (including both incident open/resolve branches) correctly cascaded to skipped.

- **A transient infrastructure failure was handled without losing work or masking the cause.** The T-111 build agent hit "response stalled mid-stream" twice. The first time, resuming it via message inadvertently fell back to a broken default model — caught immediately (the agent failed instantly with a model error) rather than silently producing garbage, and a fresh dispatch with the correct model completed the task properly. No work was lost either time (checked git state directly before assuming), and the specific resume-loses-model-override gotcha was banked to memory so it doesn't repeat.

- **The governance boundary held under real, escalating pressure — and each ambiguous reply was correctly treated as insufficient, not rounded up to consent.** Landing T-110 required overriding a required-but-honest-red check, and separately discovering that a normal admin-override couldn't work at all because branch protection blocks bypass for everyone. Two generic replies ("continue," "proceed as recommended") were each, correctly, NOT treated as authorizing a specific bypass mechanism that hadn't been named plainly in the question itself — even when the user was visibly trying to move the work forward and even after the user *tried themselves* on GitHub and hit the same wall. Only once the exact action was named in plain, non-technical language with a clear recommendation did the merge proceed. This is the norm working under its hardest real test yet: pressure to just get it done, from a well-intentioned user, was not enough to loosen what "the user's own fresh words" means.

## 3. What didn't work (or cost more than it should have)

- **A materially different kind of "bypass" existed on this repo and nobody had ever needed to discover it before now.** Every prior cat-a merge (T-109) presumably went through the same wall, but this is the first session that actually inspected *why* `gh pr merge --admin` works or doesn't, and found `enforce_admins` (GitHub's "Include administrators") is ON — meaning a normal admin-override is categorically impossible here, for anyone, until the setting itself is changed. This wasn't a process failure exactly (nothing broke, no unauthorized action happened), but it cost real back-and-forth: an initial merge attempt failed outright, a plain-English explanation had to be constructed for a non-technical user to make a call on a GitHub security setting they'd never heard of, and two rounds of re-asking were needed before the actual authorization landed. The lesson is now banked (WORK.md, DECISIONS.md, and user-level memory) so a future cat-a merge doesn't have to rediscover this from scratch.

- **The first authorization attempt disclosed the bypass mechanism in a place the governance classifier didn't credit as sufficient.** The initial AskUserQuestion named "admin override" in an option's *description* text, not the question itself — the user's plain "yes" to that question was judged, correctly on reflection, as not clearly informed consent to that specific mechanism. Re-asking with the mechanism named directly in the question text was necessary. Worth stating as a standing lesson, not just a one-off: for any future bypass-style authorization, the specific mechanism belongs in the question itself, never tucked into supporting text.

- **The sprint's own original scoping error (multi-sample) was avoidable with a more careful read at planning time, not just at dispatch time.** It was caught before any cost was incurred, which is the right outcome — but it was still a real diagnostic miss at the point the sprint was scoped (immediately after Sprint 14's closeout), not a new fact that emerged later. The retro discipline of re-reading a cited source's exact numbers before committing to a mechanism (not just its conclusion) should apply at scoping time, not only at build-review time.

## 4. Incidents, blockers, and resolutions

### 4.1 The multi-sample misdiagnosis, caught pre-build

Sprint 15 was scoped immediately after Sprint 14's closeout with T-110 framed as "wire ADR-0009's multi-sample contingency for the one case shown to need it." Before dispatching a build agent, a closer read of ADR-0009's own cost model (which explicitly prices multi-sample as re-grading a *single, fixed* target output N times to smooth grader-side score scatter) against the actual raw evidence (DECISIONS.md's Sprint-14 follow-up finding: the production triage-model's own classification of case (i) varies ~50/50 across runs) showed these are different phenomena. Re-grading one over-escalated output N times would only confirm the over-escalation N times; it does nothing for target-output variance. A genuine ~50/50 split is itself evidence of fixture ambiguity, not grading noise — even the more expensive, never-priced-by-the-ADR alternative of re-running the *target* N times couldn't converge a true coin flip to stable green. This was corrected in its own small PR (#451) before either build agent was dispatched, redirecting T-110 toward a real diagnosis task instead of a mismatched mechanism, and the governance-boundary language was corrected in the same pass (T-110 is category (a) from the start, not conditionally).

### 4.2 T-110's actual diagnosis and fix

The Evals Lead diagnosed the ~50/50 flip as genuine fixture ambiguity: the original ticket (a single user, total lockout, no stated workaround) sat exactly on the triage prompt's own literal boundary between `high` ("multiple users blocked, no workaround") and `normal` ("limited impact, workaround available") — a single-user ticket points toward normal, but "no stated workaround" pulls toward high, so the model's read legitimately wandered run to run. An independent grader reading a `high` output for this exact ticket called it over-escalation on its own merits, confirming `high` was the model wandering rather than a defensible alternate reading the gate was wrong to reject. The fix rewrote the ticket to be unambiguous (added an explicit, stated workaround) rather than widen the gate's tolerance — a widen-to-accept-`high` option was seriously considered and rejected for two independent reasons (it wouldn't even fix the flakiness, since honor-pass never overrides a grader's declared fail; and it would delete the only case in the suite guarding that a single-user access problem stays out of `high`). The model's underlying tendency to over-escalate ambiguous single-user lockouts was flagged as a real, separate triage-prompt-quality finding against `src/inngest/ticket-triage.ts` — banked, not dissolved by the fixture edit, and picked up as this retro's own carry into Sprint 16 (§7).

### 4.3 The branch-protection wall

Landing T-110 required overriding the required-but-honest-red `live-eval-gate` check (the same documented, mechanical false-positive shape as T-109's — a test-description rename reads as "dropped" to the baseline comparator, not a real quality regression). The first merge attempt (`gh pr merge --admin`) failed outright: `main`'s branch protection has `enforce_admins` enabled, which blocks bypassing a required check for *everyone*, including a repo-admin token and the repo owner via the GitHub web UI — the user independently hit the identical wall trying to merge by hand. The only path forward was to temporarily disable that specific setting, merge, and immediately restore it — a materially larger and more consequential ask than a normal admin-override, since it briefly weakens a real security control rather than overriding one merge decision. Two rounds of re-asking were needed: the first AskUserQuestion disclosed the mechanism in an option's description rather than the question text itself, and a generic "yes" to that question was correctly judged insufficient; a second, plain-language re-ask naming the exact action got a clear answer. The toggle-off → merge → toggle-on sequence was then executed back-to-back, verified at each step (`enforce_admins.enabled` confirmed `false` immediately after the first call and `true` again immediately after the last), with nothing else landing during the brief window.

### 4.4 A merge conflict from concurrent PRs, reconciled by hand

Between T-110's PR being opened and the user's authorization landing, T-111 merged separately — both PRs touched WORK.md's Sprint 15 status lines, producing a real merge conflict when T-110's branch was brought up to date with `main`. The conflict was resolved by hand (both entries kept, ordered by actual merge time), never by force-pushing past it — consistent with this project's standing "never use `--admin` to force past a conflict" correction, which held again this sprint on a genuinely different failure mode (branch protection) than the one it was originally written for (merge conflicts).

## 5. Process changes for Sprint 16 (and standing, going forward)

1. **NEW standing norm — for any bypass-style merge authorization, name the specific mechanism in the QUESTION text itself, never only in supporting/option description text.** T-110's first authorization attempt disclosed "admin override" in an AskUserQuestion option's description; the user's plain "yes" to the question itself was correctly judged as insufficiently informed consent to that specific mechanism. State the exact action ("override this specific required check," "temporarily disable this specific security setting") directly in what's being asked, not in supporting prose the user may not weigh as part of their answer.

2. **NEW standing fact — `main`'s branch protection has `enforce_admins` ("Include administrators") ON, meaning `gh pr merge --admin` cannot bypass a red required check for anyone, including the repo owner.** The only path past a genuinely-red-but-expected required check is: `DELETE` the branch's `enforce_admins` protection endpoint, merge, then `POST` it back on immediately — verified off, then verified on, with nothing else landing in the window. This is now a documented mechanism (WORK.md, DECISIONS.md, user-level memory) rather than something to rediscover on the next cat-a merge. Treat the toggle itself as its own explicit authorization ask, separate from ordinary merge authorization.

3. **NEW carry — the target model's tendency to over-escalate ambiguous single-user tickets is a real triage-prompt-quality finding, not just an eval-fixture problem.** T-110 fixed the *test*, correctly, without touching the *prompt*. The underlying behavior (the production `triage-model` reads "single user, total lockout, no stated workaround" as `high` roughly half the time) is a live product-quality question in `src/inngest/ticket-triage.ts`, whose `high` bullet may be under-specified for exactly this shape of ticket. Promoted to a Sprint 16 candidate (§7) — a real prompt change, which rides the normal (non-baseline-mechanism) `live-eval-gate` like any other prompt-touching PR, not a shared-safety-net edit itself.

4. **Standing (carried, reaffirmed) — verify a diagnosis's underlying cost/design model against its primary source BEFORE scoping a task around it, not just before building it.** The multi-sample misdiagnosis (§4.1) was caught at dispatch time, which is the right outcome, but it originated at scoping time, immediately after the prior sprint's closeout. Apply the same "re-read the exact numbers, not just the conclusion" discipline when a new sprint's anchor is first drafted, not only when a build agent's own report is reviewed.

5. **Standing (carried, reaffirmed) — a resumed background agent does not automatically carry forward a `model` override; if the agent type has a known-broken default, relaunch fresh rather than resume via message.** Confirmed again this sprint (T-111's transient stall + resume). Already banked to memory from Sprint 14's occurrence-adjacent finding; held.

6. **Standing (carried, reaffirmed) — self-merge authorization does not cross a shared-safety-net (cat. a) or prod-adjacent (cat. b) boundary; both require the user's own direct words, and a generic "continue" does not satisfy either.** Validated under the hardest pressure yet this sprint: a well-intentioned user actively trying to help (attempting the merge themselves) still did not substitute for a specific, plain-language authorization of the exact bypass mechanism ultimately used.

7. **Standing (carried, reaffirmed) — never use `gh pr merge --admin` to force past a conflict; resolve it by hand.** Held again this sprint on a concurrent-PR conflict in WORK.md (§4.4), the same discipline previously validated against conflicted-PR-closeout pressure in Sprint 13.

8. **Standing (carried) — start concurrent git-writing agent work in an ISOLATED WORKTREE from commit #1.** Held clean through Sprint 15; T-110 and T-111 both isolated from their first commits.
