## 2. What worked

- **Defense in depth caught five distinct real problems before any reached production — this is the sprint's headline.** T-98's path to go-live is the cleanest demonstration this team has of layered verification earning its keep. In sequence: (1) the **harness's own safety classifier** correctly blocked the first build attempt for touching production credentials/data in a new way without explicit authorization; (2) a **Security Lead design review** — *before* implementation — found the original plan rested on a false premise (a claimed "T-51 FreeScout REST-API precedent" that does not exist; ADR-0003 Option C explicitly rejected FreeScout's REST API) *and* a real cleanup flaw (the original per-run insert-then-FK-null cleanup could have let a real customer's action mint a duplicate production ticket), replacing both with the permanent-sentinel pattern (Amendment A1) and a new narrow role instead of `OPS_HUB_APP_LOGIN_URL` in CI (Amendment A2); (3) the **SC1–SC10 negative-test suite** verified the role/RLS scoping on a real all-four-green run; (4) a **final Security Lead sign-off** caught BC1 — the incident-alert step's `if:` had no `failure()`/`always()` status function, so GitHub's implicit `success()` left SC8's paging path as dead code that would never fire on a real failure; (5) a **mid-fix re-verification** of BC1 caught a *second* real bug — the reset-step's row-count check was a false-pass that could have fired a real Inngest event during a supposedly side-effect-free verification run. Each layer caught something the layers before it did not. A monitor built carelessly is worse than no monitor; this one was built with its own failure modes as first-class test cases, and every safety layer that could catch a problem did.

- **The monitoring investment paid for itself on its first real opportunity — this is why we built it.** FQ-76 is the proof of the T-97/T-98 thesis. The T-97 internal-auth monitor (Sprint 9) fired, and the *actual* production break behind it — ops-hub-prod's `LITELLM_URL` gone stale after FQ-70's redeploy — was **caught and fixed before any customer ticket was affected.** Contrast with FQ-69 (Sprint 8): the *same class* of break sat undetected for 3.6 days while 70% of production tickets jammed, precisely because the old `/health` check structurally could not see it. This time the live tickets table showed the *inverse* signature — 20 resolved + 1 responded, **0 stuck** — because the monitor bought the team the chance to fix it in the gap before the next customer email arrived. A monitor that catches a real break pre-impact, on its first genuine firing, is the entire return on the Sprint 9/10 monitoring spend, realized.

- **The two verification bugs found in SC1 were harness bugs, not security bugs — the taxonomy held under pressure.** SC1's `verify-e2e-monitor-role.yml` reported (b) FAIL and (c) FAIL on its first real run. Both were bugs in the *verification harness*, not in the DB security: (b) a missing `ON_ERROR_STOP` meant psql exited 0 on an RLS violation, tripping the "INSERT SUCCEEDED" branch before the confirmation grep could run; (c) captured the `INSERT 0 1` command tag instead of the `RETURNING` UUID. The DB was rejecting the bad write correctly all along (the raw log showed "new row violates row-level security policy") — the harness was mis-reading it. This is the same class as the T-84→T-88 harness-bug saga, and the team named it correctly again: the security behaviour was right; the test was wrong. The fixed run then proved the role green-for-the-right-reason (a genuine matched-then-rejected UPDATE, not a silent `UPDATE 0` no-op), not merely "four successes."

- **The design-review-before-build discipline from Sprint 9 §5.1 caught real risk again, this time on the anchor.** The Security Lead design review of T-98 (before a line was implemented) killed two genuine problems — the false-precedent premise and the duplicate-prod-ticket cleanup flaw — and reshaped the whole build (Amendments A1/A2, SC1–SC10). Had T-98 been built to its original brief, it would have shipped a monitor capable of minting a duplicate production ticket from a real customer's action. Review-before-implement earned its keep on the sprint's most consequential task.

- **Track B was clean, disciplined execution.** T-99 grew each eval N=4 → N=9 with *real prompt-grounded* coverage (security-breach→critical, injection resistance, redaction-defeat, no-hallucination on escalated-unresolved, non-English robustness — not filler), proved the 15 new cases through the live gate with all T-91 calibration guards green, re-captured a 27/27 baseline, and persisted the row. It also self-corrected one instance of an unnecessary admin-bypass merge command it had reached for. T-100 vetted `meta/llama-3.3-70b-instruct` for both functions at 9/9, and — notably — *noticed mid-task* that T-99 (running concurrently) had grown the suite, then re-validated against the current N=9 rather than reporting the stale N=4 result. Riding the now-live gate made both low-risk; the gate had more to catch and caught nothing, which is the point.

- **An agent self-corrected an unnecessary privilege escalation, unprompted.** T-99's Evals Lead used, then walked back, an unauthorized admin-bypass merge command it did not need. Same family as Sprint 9's T-96 "refused to force a false pass" — the reflex to reach for the maximal-privilege shortcut is being caught and corrected by the agents themselves, not only by review.

## 3. What didn't work (or cost more than it should have)

- **FQ-76 is the THIRD instance of the same "redeploy orphans the internal URL" failure class — the band-aid ceiling is reached.** T-71 (Sprint 6), FQ-69's `LITELLM_URL` sub-issue (Sprint 8), and now FQ-76 (Sprint 10) are all the same shape: a litellm-prod redeploy rotates its internal Docker container suffix, ops-hub-prod's pinned `LITELLM_URL` is not re-synced, and the app's internal hop breaks (`getaddrinfo` fails → fetch throws → "unreachable"). Each time it has been re-aligned by hand via `fix-ops-hub-prod-litellm-url.yml`. That is a reactive patch applied three times; the Tech Lead has explicitly flagged that a durable fix — a stable internal address, a post-redeploy re-sync hook, or app-side service-name resolution — is now warranted over a fourth manual re-align. **Committed as the Sprint 11 anchor (ADR-first)** (§5, §7). *Note the honest classification:* this is the **stale-URL class (T-71)**, distinguished at the handler level from FQ-69's master-key-rejection class (401 `token_not_found`) — see the caution in §4.2.

- **FQ-76 was initially mis-classified, and the correction matters for what stays open.** T-98/PR #407 correctly *flagged* the live `/health/litellm-internal` degraded incident but filed it (implicitly) under the "provider-credential divergence" carry — the master-key/Anthropic-key rotation class whose Sprint-9 revisit trigger was "a third instance of provider-credential drift." The Tech Lead's read-only diagnosis corrected this: FQ-76 is the *URL-suffix* class, not the credential-divergence class. Two consequences worth holding: (a) the "3rd instance / band-aid ceiling" count is T-71 → FQ-69-URL → FQ-76, all URL-class; (b) **the provider-credential-divergence carry stays open and uncommitted** — its trigger did *not* actually fire, it was mistagged. Do not retire it or fold it into the URL fix. A near-miss on the carry-forward ledger, caught by a careful second look rather than accepted at first tag.

- **The T-98 verification record itself had to be corrected mid-sprint — an optimistic "chain proven" claim was walked back against real run data.** A prior session's entry asserted the full event-dispatch → triage → trace chain was "proven end-to-end." On re-examination against `gh run view` output, that run's own conclusion was `failure` and its LangFuse-trace step was `skipped` — never ran. The claim was likely written from the pre-fix GUC bug's misleading non-empty poll artifact rather than the run's actual outcome. It was corrected plainly across WORK.md, DECISIONS.md, and the deploy record. The lesson is a good one to bank: **read the run's actual conclusion, not its most convenient intermediate output** — the same discipline the eval-gate calibration guards enforce structurally, applied by hand here.

- **T-98 needed manual provisioning that no CI path could do, extending its tail.** The first genuine `mode=live` run required, in order: a FreeScout test conversation created via the browser UI (FreeScout's REST API is disabled-by-default — the false premise from §4.1), a one-time sentinel-ticket DB-row insert built as a **brand-new workflow mid-session** (the monitor's reset-only design deliberately never inserts), and a founder-run migration + Inngest-key mint (FQ-75). None of this was avoidable given the design constraints, but it is honest to record that the anchor's last mile was a sequence of manual, founder-gated, and net-new steps rather than a clean dispatch — and that an initial `E2E_SENTINEL_TICKET_ID` was set to the FreeScout conversation number instead of the ticket UUID (caught immediately and loudly by the reset step's `::uuid` cast — a fail-loud design working).

- **Operational observation: the "response stalled mid-stream" subagent failures recurred again — now across two sprints.** Several subagent runs this sprint hit the same `API error: response stalled mid-stream` pattern first noted in Sprint 9's retro (§5.3). As before, every run was resumed or retried and **no work was lost.** It is **not** root-caused and **not** yet actionable. Recording it again because it has now persisted across two consecutive sprints — a data point worth keeping so a future PM has continuity if it ever worsens or a stall lands after a partial commit. Tracked, not owned (§5).

## 4. Incidents, blockers, and resolutions

### 4.1 T-98 — five verification layers, five distinct real problems caught, all pre-production

The anchor's build is the sprint's defining story. It is worth laying out the full sequence because each layer caught something distinct, and the composite is defense-in-depth working exactly as designed — not a build that went badly, but a build whose safety controls all fired.

1. **Harness safety classifier (build-time).** The first build attempt was correctly blocked by the harness's own safety tooling for touching production credentials/data in a new way without explicit authorization. Not a false positive — the *right* stop before an unreviewed prod-data-touching monitor got written.

2. **Security Lead design review (pre-implementation).** Found two real problems in the *original plan*, before any code: (a) a **false premise** — the brief asserted a "T-51 FreeScout REST-API precedent" for creating test conversations; verified FALSE (ADR-0003 Option C explicitly rejected FreeScout's Api module as disabled-by-default; T-51 actually sent a real inbound email); (b) a **real cleanup flaw** — the original per-run insert-then-FK-null cleanup could have let a real customer's action mint a duplicate production ticket. Both were replaced: Amendment A1 (permanent sentinel-ticket pattern, immune to reopen-duplication via the existing `freescout_conversation_id` UNIQUE + ON CONFLICT dedup) and Amendment A2 (a new narrow `e2e_monitor` role instead of `OPS_HUB_APP_LOGIN_URL` in a scheduled CI context). Verdict: APPROVED WITH CONDITIONS (2 amendments + SC1–SC10).

3. **SC1–SC10 negative-test suite (implementation verification).** `verify-e2e-monitor-role.yml` proved the DB role's scoping dispositively (prod-tenant SELECT → 0 rows; prod-tenant INSERT → RLS violation; synthetic round-trip → succeeds; tenant-escape UPDATE → RLS violation). Its first real run surfaced **two harness bugs** (missing `ON_ERROR_STOP`; command-tag captured instead of the `RETURNING` UUID) — **not security bugs**; the DB was rejecting the bad writes correctly the whole time, the harness was mis-reading the result. Fixed harness-only; the corrected run proved the role green-for-the-right-reason (a genuine matched-then-rejected UPDATE, not a silent no-op). The `e2e_monitor` credential was then APPROVED at final sign-off.

4. **Final Security Lead sign-off (BC1 — a real monitor bug).** Split verdict: the credential APPROVED, but the *workflow* held on ONE BLOCKING CONDITION — the "on failure, open incident" step's `if:` had no `failure()`/`always()`/`cancelled()` status function, so GitHub's implicit `success()` left the step SKIPPED on every real failure. SC8's status-page paging path was **dead code** — the monitor would have run for months and never paged on a real break. Fixed (PR #413, added `failure()`), and proven with three real dispatches: `mode=simulate-failure` forced a genuine failure and confirmed the incident step now *runs* and opens a real status-page incident, then resolved it; `mode=live` confirmed dormancy unchanged.

5. **Mid-fix re-verification (a second real monitor bug).** Proving the BC1 fix surfaced that the "reset sentinel ticket" step's own 0-rows-matched detection was a **false-pass** (psql's `UPDATE 0` command tag is non-empty text, so the step's `-z` emptiness check never caught a no-op). Initially mis-scoped as harmless — on reflection it undermined the simulate-failure mechanism itself: post-provisioning, the fake test ticket would false-pass reset, open the dispatch gate, and **fire a real Inngest event with the real sentinel id during a supposedly side-effect-free verification run.** Fixed (PR #415, check for the literal `UPDATE 1` tag — the same technique `verify-e2e-monitor-role.yml` check (d) already used) and re-proven: simulate-failure now fails *at* the reset step, dispatch/poll/langfuse correctly skipped, no real Inngest POST fires.

**Go-live.** FQ-75's both founder actions landed (migration + password, dedicated Inngest key), SC9's FreeScout test conversation (#34) was created, the one-time sentinel-ticket INSERT was built as a new workflow (`provision-e2e-sentinel-ticket.yml`), and the **first genuine `mode=live` end-to-end run — [run 29209542703] — succeeded in full**: reset → real Inngest dispatch → polled to `state='responded'` → LangFuse trace asserted → green, no incident opened. This is the first real proof of the complete downstream chain the monitor exists to watch, and it retired the prior "residual watch item" (a real `UPDATE 1` reset had never fired) as a byproduct. T-98 is now live, scheduled 6-hourly, and structurally safe.

**The generalizing point:** the two harness bugs (SC1) were test-tooling defects; BC1 and the reset false-pass were *real monitor* defects that would have shipped a monitor that could not page and could fire spurious events. **Both real monitor bugs were caught only by a `mode=simulate-failure` dispatch** — actually exercising the failure path, not reviewing the YAML. That is the process change §5 proposes.

### 4.2 FQ-76 — the T-97 monitor caught a real production break before any customer was affected

**How it was found:** T-97's `monitor-litellm-internal-auth.yml` (built Sprint 9) fired for three consecutive days; T-98/PR #407's finalization flagged the live `/health/litellm-internal` degraded incident, and the Tech Lead ran a read-only diagnosis to root-cause it.

**Root cause — the T-71 stale-URL class, not the credential-divergence class (this distinction matters):**
- **Failure shape:** `/health/litellm-internal` → HTTP 503 `{"status":"degraded","litellm_internal":"unreachable"}` — the `healthLitellmInternal.ts` CATCH block (fetch threw on the internal hop), **not** the `auth_rejected` (401 `token_not_found`) branch that was FQ-69's exact signature. Not the master-key class.
- **AI service is UP:** `/health/litellm` (external URL, LiteLLM's own key) → HTTP 200. Only the app's *internal* Docker-network hop was failing → an addressing problem, not a dead service or a bad key.
- **Stale suffix, proven:** ops-hub-prod's `LITELLM_URL` suffix `-132650269773` (the value FQ-69's fix set on 2026-07-09) vs. the now-running litellm-prod container `-140935289661`.
- **The "how" is established** (which FQ-69 never pinned down): the founder's FQ-70 manual redeploy of litellm-prod (Anthropic-key swap, re-verified 14:11:44Z) rotated the suffix; the monitor went red 14:12:52Z, ~1 minute later. A documented founder action's side-effect, not silent drift.

**Zero customer impact — verified, not assumed:** the live tickets table showed 20 `resolved` + 1 `responded`, **0 in `new`** — the exact *inverse* of FQ-69's stuck signature (14 in `new`, owner_agent NULL). The break had not bitten because no new ticket had arrived since the redeploy; the next one would have stuck until the URL was re-aligned. The monitor caught it in the pre-impact window — the outcome T-97 exists to produce.

**Resolution:** FQ-76 filed (founder-gated, FQ-69 plain-language format); founder authorized `fix-ops-hub-prod-litellm-url.yml` with the re-confirmed container name; verified with the *real* check (`/health/litellm-internal` → 200), not the generic `/health` (FQ-69's blind spot). **Filed and resolved same day; no customer ticket ever affected.**

**Two Tech Lead follow-ups (not this sprint, now scheduled — §5, §7):** (1) a durable fix for the redeploy-orphans-URL class (band-aid ceiling reached at 3 instances) — a Tech Lead architecture item, Sprint 11 anchor; (2) a 2–3-consecutive-fail monitor threshold so transient blips (the 07-10/07-11 self-healing hiccups that made "third consecutive day" look worse than it was) stop crying wolf while sustained breaks still page.

## 5. Process changes for Sprint 11 (and standing, going forward)

1. **NEW standing norm — prove a monitor's failure/alert path actually fires (`mode=simulate-failure`) before trusting it live.** This sprint produced the justification: **two** real monitor bugs (BC1's dead-code incident step; the reset-step false-pass that would have fired a real event during a side-effect-free run) were caught *only* by a simulate-failure dispatch, not by YAML review or a happy-path run. A monitor's whole value is what it does when something breaks; a monitor whose failure path has never been exercised is unverified precisely where it matters most. Make it explicit: no monitor/alerting workflow goes live (schedule enabled) until a `mode=simulate-failure`-style dispatch has proven the failure branch runs, opens its incident, and — critically — has no spurious side-effects on the way. T-97 already carried this mechanism; T-98 adopting it is how both its real bugs surfaced. Cheap, and a perfect hit rate this sprint.

2. **Standing (carried, reaffirmed) — health/monitoring must exercise the REAL path.** FQ-76 is the *third* live confirmation of this lesson (after FQ-69 and FQ-70): the generic `/health` stayed green through the whole break; only `/health/litellm-internal`, which exercises the real internal hop, went red. The all-clear signal after any such fix is the *real-path* check plus the monitor auto-resolving its own incident — never the generic health check. Reaffirmed, not new.

3. **Standing (carried, reaffirmed from Sprint 9 §5.1) — any credential/secret-store change gets a Security Lead review BEFORE implementation.** Validated again on the anchor: T-98's design review (before code) caught the false-precedent premise and the duplicate-prod-ticket cleanup flaw. The harness's own safety classifier flagging the prod-data-touching build is part of this control, not friction — treat a flag as the *start* of the review. Perfect hit rate now spans two sprints.

4. **Track (carried, now 2-sprint watch item) — subagent `response stalled mid-stream` transient failures.** Recurred again this sprint; no work lost (resume/retry recovered every time); **not root-caused, not actionable.** Now persisting across two consecutive sprints. Escalate to an actual investigation only if it worsens materially or a stall ever lands *after* a partial commit (where recovery would not be free). No owner, no task — a watch item, tracked for continuity.

5. **Standing (carried, reaffirmed): calibration-guard the harness, read-before-mutate, deploy-verify SHA discipline, read the actual run conclusion.** All in force. The last one earned a fresh reminder this sprint (§3): the corrected "chain proven" claim was written from a misleading intermediate artifact rather than the run's real outcome.
