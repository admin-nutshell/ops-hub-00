## 2. What worked

- **The eval-gate build closed the oldest live-vs-record drift on the books — the one at our own front door.** Every prior sprint chased drift between what a record claimed and what the live system did: a migration file that said a policy existed while the DB had lost it (Sprint 7/8), a health check that said green while 70% of tickets were stuck (FQ-69). The single most-repeated instance was in CLAUDE.md itself: the "Eval-gated" standing constraint asserted a live >95% gate since T-58/T-79, while CI ran schema validation only — a self-drift the team documented honestly (T-86's interim caveat) precisely because it couldn't yet fix it. Sprint 9 fixed it. `live-eval-gate` is now a required, merge-blocking check that runs the real evals against each function's production LiteLLM target alias, grades them with a **distinct** judge alias, blocks on any regression versus the last green baseline, and has proven all of this on a genuine graded run (12/12, first real row in `eval_gate_runs`). T-95 then made the doc honest again. This is the first time the team retired a live-vs-record drift class by building the missing enforcement rather than by documenting the gap — and it retired the class the team itself lived inside.

- **The ADR-not-build discipline from Sprint 8 paid off exactly as intended.** Sprint 8 deliberately produced ADR-0007 as a design spike and deferred the build, sizing it *medium* with four named build conditions (C1–C4). Sprint 9 executed against a settled design: C1 (scoped key gates the auto-trigger) shaped the T-89/T-90/T-93 ordering; C2 (sibling workflow) kept the hermetic `validate` loop untouched; C3 (per-test baseline) drove the JSONB decision; C4 (fork/non-match wedge) was the load-bearing call T-94 had to get right. None of these were discovered mid-build — they were on the map. A medium build from a settled design is what "deferral that is scheduled, scoped, and named is not a punt" looks like in the ledger.

- **The T-84 calibration lesson became a permanent structural defense, and it was proven, not asserted.** ADR-0007 §5 turned Sprint 8's painful harness bug (a confident, specific, *wrong* 25%) into three guards. T-91 built them and — crucially — proved every branch against **real promptfoo-0.121 output** with only the model call stubbed: a token collapse toward user-only tokens (the exact T-84 signature) now hard-errors (exit 2); a rubber-stamp grader that passes a must-fail canary hard-errors (exit 4); a `judge == target` shortcut fails fast before any metered call (exit 3). The next silent-measurement failure, in *either* direction, is now a loud error instead of a number nobody should have trusted.

- **The gate's own build surfaced real bugs the moment its adjacent paths ran for real — and each was caught before it could matter.** T-94's grader≠target re-capture found that `capture-eval-baseline.yml` was globbing canary results into the product baseline (18 tests vs the gate's 12 → a *false* GATE FAIL against any fresh baseline). T-93's final verification found that `psql -c` never interpolates `:'var'` bind variables — a bug live in `persist-eval-run.sh`, the production persist path, that would have failed its first real run. T-92's advisor-caught seam fix found that `compare` read only the nested baseline shape, so once T-93 fed it the flat DB-row shape it would have seen zero baselined tests and **passed everything** — the swap-masking dead-gate the whole track exists to close, reopened one seam over. Each was found and fixed *before* the gate became required. A gate you build carelessly is worse than no gate; this one was built with its own failure modes as first-class test cases.

- **Read-before-implement on every credential change caught real risk three times (see §3/§4.2).** This is worth calling out as a *win*, not just a process note: each of the three credential-provisioning cycles this sprint was gated by a fresh Security Lead review before implementation, and each review caught something the "just provision it" path would have missed. Defense in depth earned its keep three times in one sprint.

- **An agent refused to force a false pass when the sanctioned path was blocked.** T-96 could not be completed on the scoped-key path as first attempted — the only structurally-sound candidate alias was deliberately out of the eval key's scope, and the master key was forbidden by design. The Evals Lead recorded the structural blocker in full (DECISIONS.md 2026-07-11) rather than reaching for the master key, forcing a partial, or fudging a pass. That honesty is exactly what the eval gate is *for*, applied to the eval gate's own construction (§4.3).

## 3. What didn't work (or cost more than it should have)

- **FQ-70 — a production resilience gap was, once again, found by side-effect rather than by a monitor.** T-90's provisioning smoke-tests, not any alert, surfaced that litellm-**prod**'s `fallback-model` was broken (a rejected Anthropic key, HTTP 401 — a distinct fault from staging's credit-exhaustion). This is the **same shape as FQ-69**: a customer-impacting resilience gap (if the primary model ever failed in prod, the fallback would also fail) that had been silently broken with nothing watching for it, surfaced only because a routine task happened to look in the right place. The pre-discovery duration is genuinely unknowable from the record — which is itself the point: nothing was measuring it. T-97 closes the *internal-auth-hop* class of blind spot, but the FQ-70 fallback path is a *different* untested route, and it reinforces the same standing lesson — health signals that don't exercise the real path can be green while the real path is broken.

- **FQ-70's staging half didn't just risk the product — it delayed the sprint's own flagship deliverable.** litellm-**staging**'s `fallback-model` (the designated *judge* alias) was Anthropic-credit-exhausted for most of the sprint. Because the gate is designed to run **grader ≠ target** (a T-84 lesson — never let the judge be the thing under test), the eval gate could not run its own intended fully-graded configuration until FQ-70 cleared. Every "proof" of the gate up to 2026-07-12 was against real-promptfoo-output-with-stubs (T-91), a `judge == target` placeholder, or a neutral-skip — all legitimate, but none the real thing. The first genuine grader≠target graded green (T-94, run 29196855171) only became possible *after* the founder topped up billing. So a single external billing fault sat on the critical path of both a customer-facing resilience feature and the sprint anchor. Worth internalizing: an external dependency (provider billing) with no local buffer is a single point of failure for anything that depends on it, including our own tooling.

- **The DB-persistence half of the gate took four founder round-trips (FQ-71 → FQ-72 → FQ-73) and a security ruling to land one feature.** Persisting each run's result to `eval_gate_runs` required, in sequence: a nullable column (FQ-71), a new least-privilege role (FQ-72), and a password + secret provisioning (FQ-73), plus a Security Lead ruling that rejected the "one-click GitHub workflow" shortcut (it would have parked a service-role-class DSN as a persistent Actions secret). Every one of those calls was *correct* — the least-privilege outcome is genuinely better than reusing an existing credential (§4.2) — but it's honest to record that the persistence feature was the long pole, gated repeatedly on founder-only `service_role` actions (non-negotiable #3), and that the gate's *core* value (catching a regression, blocking a bad PR) never depended on any of it. The persistence is history/observability, and it was correctly built as non-fatal to the gate decision throughout.

- **Operational observation: transient "response stalled mid-stream" subagent failures recurred this session.** Several subagent runs hit an `API error: response stalled mid-stream` early in their run, before any commits. In every case the run was resumed or retried fresh and **no work was lost** — but it happened often enough this session to flag as a pattern. It is **not** root-caused and **not** yet actionable; recording it here so that if it worsens (or if a stall ever lands *after* a partial commit) there is a prior data point rather than a cold start. Tracked, not owned (§5).

## 4. Incidents, blockers, and resolutions

### 4.1 FQ-70 — production fallback broken (invalid key) + staging judge exhausted (billing); one fault, two halves, both on critical paths

**How it was found:** T-90's virtual-key provisioning smoke-tests, not a monitor. The positive test against the judge alias (`fallback-model`) failed — on litellm-staging with an Anthropic *credit-exhaustion* 400, which prompted a read-only check of whether prod shared the problem.

**Two distinct faults, kept distinct:**
1. **litellm-staging `fallback-model` — Anthropic credit exhausted (HTTP 400).** Blocked the eval gate from running its intended grader≠target configuration (the judge alias was the exhausted one). An availability/billing fault, external, not a key-scoping defect (T-90's key *is* correctly scoped to include the alias).
2. **litellm-prod `fallback-model` — invalid Anthropic key (HTTP 401), a separate registration.** A genuine production resilience gap: primary-model outages in prod had no working fallback. Also carried the Coolify duplicate-`ANTHROPIC_API_KEY`-row footgun (count only checked, values never read — the "read a count, not the credential" discipline held under investigation pressure).

**Resolution (both halves, 2026-07-12, founder actions):** founder topped up the Anthropic account (staging half → the eval gate ran its first real fully-graded check as a *direct* result, run 29195017036, GATE PASS, DB write confirmed) and rotated litellm-prod's `ANTHROPIC_API_KEY` in Coolify + redeployed (prod half → re-verified `POST /chat/completions (model=fallback-model)` → HTTP 200, duplicate rows cleared to exactly one). Production's fallback path is fully restored.

**Filed as FQ-70 (non-blocking)** per the customer-impacting-incident escalation criterion — a founder billing/credential decision, not a team defect. The lesson (§3): an external billing dependency with no local buffer is a single point of failure for anything downstream of it — including our own eval tooling.

### 4.2 Three credential-provisioning cycles, each reviewed before implementation — defense in depth, working as designed

This sprint provisioned or re-provisioned scoped credentials three times, and every time the change was gated by a fresh Security Lead review *before* implementation — not rubber-stamped, not reviewed-after. This is recorded as a **good** outcome: the review-before-implement norm (and the harness's own safety tooling flagging secret-store mutations for that review) caught a concrete, distinct risk each time.

1. **T-90 — the scoped LiteLLM virtual key.** Review verified the actual `/key/generate` payload and the negative scope-boundary proof (HTTP 403 on an out-of-scope alias), and confirmed the key holds no management capability (cannot mint keys or touch routing) — blast radius ≈ $7/30d of staging completions, an availability nuisance, not data/config exposure. Three non-blocking observations (query-string key exposure to our own logs; empty `soft_budget` readback; no key expiry) recorded for follow-up, not treated as blockers.

2. **T-93 — the DB-persistence credential.** The review **rejected** reusing `SUPABASE_STAGING_DB_URL` (an owner/RLS-bypass role — reusing it in an auto-triggered PR job would have exposed the whole DB, including customer tickets, if a malicious PR ever leaked it) and specified a **new least-privilege `eval_gate_ci_writer` role**: LOGIN/NOBYPASSRLS/NOINHERIT, `GRANT INSERT ON eval_gate_runs` only, **no SELECT**, RLS `with check (project_id is null and run_type='llm_rubric')`. It later also rejected the "one-click GitHub workflow" password-provisioning shortcut (persistent service-role-class Actions secret + `workflow_dispatch` inputs shown unmasked) in favour of a founder single-paste. The final verification (2026-07-12) caught a real harness bug (`psql -c` doesn't interpolate `:'var'` binds) that was live in the production persist path.

3. **T-96 — re-widening the eval key for KB Learn.** The review caught the **NIM $0-metering gap**: `meta/llama-3.3-70b-instruct` routes via NVIDIA and may price at $0/unknown in LiteLLM's cost map, meaning the `$7 max_budget` would *never trip* for that alias — the T-90 review's "$7-bounded" claim would have been silently false for one of three aliases. Closed with an added `rpm_limit=60`, plus the discipline of a **re-issue** (new key value + revoke the old alias) rather than an in-place update. Old key confirmed gone via `/key/list`.

**The generalizing point:** none of these was a process failure or friction to be optimized away. Each review found a real hole a fast path would have shipped. The norm proposed for Sprint 10 (§5) is to make this explicit and standing.

### 4.3 T-96 — the sanctioned path was genuinely blocked, and the agent said so instead of forcing a pass

**What happened:** T-96 (widen `kb_learn` to a vetted second alias) hit a real structural wall. Only three aliases were registered on litellm-staging; of them, `triage-model` was the existing pin (no second alias), `fallback-model` was FQ-70-blocked *and* was the designated judge (a grader≠target conflict), and `meta/llama-3.3-70b-instruct` — the only structurally-sound candidate — was *deliberately out of the scoped eval key's model list* (dispositively proven by T-90's HTTP 403 negative test). Reaching it needed the master key, which the whole ADR-0007 design forbids.

**What the agent did:** recorded the blocker in full (DECISIONS.md 2026-07-11) — no `model-allowlist.ts` change, no eval run against a non-existent candidate, **no master-key workaround** — and laid out three clean unblock options, correctly declining to file a founder escalation (an inter-agent technical dependency on key scope, not a business decision).

**Resolution:** option 1 was taken cleanly — Security Lead reviewed the scope-widen (C1–C7, §4.2), Production Manager re-issued the key, and the Evals Lead then ran the real vetting eval: `meta/llama-3.3-70b-instruct` passed **4/4 (100%)**, grader≠target, all calibration guards green, and was added to the allowlist with the required invariant rewrite (per-function vetting: included for `kb_learn`, still excluded from `triage`/`respond`). This is the eval gate's own honesty principle — don't claim a pass you can't stand behind — applied to building the eval gate. It's in §2 as a win because it is one; it's here because "the first attempt was blocked" is also the honest record.

### 4.4 The C4 wedge — the one architecture call T-94 had to get exactly right

**What it was:** registering a *path-filtered* `pull_request` workflow as a required check would have wedged **every** non-matching PR (docs, dashboard, infra — the majority) forever at "Expected — waiting for status to be reported," because a path-filtered trigger creates *no* check run at all on a non-matching PR. A naive registration would have jammed the whole repo.

**The fix:** move the ADR §3 path filter *off* the trigger and *into* an internal step-level gate — the job runs on every PR and always reports the `live-eval-gate` context, while the metered LLM work stays gated to prompt-surface changes (fail-safe: run the gate if the diff can't be read). Proven live: PR #394 (non-prompt path) neutral green-skipped in ~6s under the same context; a throwaway prompt-touching PR (#396) ran the full metered gate to a real GATE PASS. Both the neutral-skip and the metered `should_run=true` detection paths were proven *before* registration, closing the worst failure mode (a detection miss that neutral-skips green on a real prompt PR and *satisfies* branch protection unevaluated).

## 5. Process changes for Sprint 10 (and standing, going forward)

1. **Standing norm — any credential or secret-store change gets a Security Lead review BEFORE implementation, not after.** Validated three times this sprint (§4.2), each catching a distinct real risk (owner-role reuse; the NIM $0-metering cap gap; the unmasked-`workflow_dispatch`-input footgun). Make it explicit and standing: no scoped key, DB role, or secret is provisioned/re-issued until a fresh Security Lead review of the concrete payload/scope has signed off, with a T-90-style negative-test readback as the proof of scope. The harness's own safety tooling flagging these mutations for review is part of the control, not friction to route around — treat a flag as the *start* of the review, never a hurdle to clear. This is cheap (the reviews were same-session) and it has a perfect hit rate so far.

2. **Standing — health/monitoring must exercise the REAL path, reaffirmed by FQ-70.** FQ-69 (Sprint 8) established this for the internal auth hop; T-97 built the monitor for it. FQ-70 is the *same lesson on a different route*: the prod fallback path had no probe and was silently broken. Any credential-or-provider-dependent path the product relies on for resilience (fallbacks included) should be exercised by *something* that would go red when it breaks — see the Sprint 10 synthetic-E2E-monitor scope (§7) for the next increment. Corollary: an external dependency (provider billing) with no local buffer is a single point of failure for everything downstream, including our own tooling — worth flagging in any design that leans on one.

3. **Track — subagent transient-stall pattern (`response stalled mid-stream`).** Recurred noticeably this session; no work lost (resume/retry recovered every time); **not root-caused, not yet actionable.** Recorded so a future PM has a prior data point. Escalate to an actual investigation *only if* it worsens materially or a stall ever lands after a partial commit (where recovery would not be free). No owner, no task — a watch item.

4. **Standing (carried, reaffirmed): calibration-guard the harness, read-before-mutate, deploy-verify SHA discipline.** ADR-0007 §5 guards are now built and enforced (T-91). Read-only-first held through the FQ-70 investigation (§4.1). The Sprint 7 deploy-verify SHA-assertion discipline continues to apply to any redeploy. All in force.
