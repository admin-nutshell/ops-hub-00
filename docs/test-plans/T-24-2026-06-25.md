## Scope

Validate the Sprint-2 ticket pipeline at the **DB + RLS + state-machine seam**:
`new → triaged → responded`, plus FreeScout-intake dedup. Function internals
(classification, draft, dispatch, retries, LangFuse) are covered by the unit
suites and are explicitly **out** of this plan.

## Auth model (non-negotiable)

All assertions and state-machine writes run as `ops_hub_app_login` (the
non-`BYPASSRLS` login role) so RLS genuinely engages — same discipline as T-18.
`service_role` is used **only** for project/tenant setup and fixture teardown
(tables with no app write policy; `tickets` has no app delete policy). Asserting
state through `service_role` would pass vacuously and is forbidden here.

## Edge cases covered

- **Empty/null body:** `billingQuestion` fixture and null-body handling exercised via insert.
- **Multi-tenant isolation:** every read/write is tenant-GUC-scoped; the login role re-read in Scenario 1 is a positive control that RLS returns the owning tenant's row. Cross-tenant negative isolation is owned by T-18 (`rls-isolation.test.ts`) and not duplicated here.
- **Dedup boundary:** globally-unique timestamp-derived `freescout_conversation_id` avoids collision with real staging rows (conv ids 6, 7).
- **Idempotency / no state corruption on failure:** T-23 unit tests assert the respond error path leaves `triaged` intact.
